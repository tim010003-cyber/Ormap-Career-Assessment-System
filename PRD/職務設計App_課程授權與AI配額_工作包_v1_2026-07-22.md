## 1. 為什麼需要後端

目前 API Key 在使用者自己電腦的本機代理（`127.0.0.1:8788`），線上訪客連不到。要讓真 AI 上線，Key 必須移到伺服器端，而 Key 一旦上線就會產生費用，所以**配額控制必須跟 AI 同時上線，不能分兩期**。

Key 絕對不能出現在前端、localStorage、版控或日誌。

## 2. 成本結構（實測）

跑完一個完整案例的呼叫次數：

| 環節 | 次數 | 計入配額 |
|---|---|:---:|
| 問題釐清（開場＋五章） | 6 | ✅ |
| 需求界定 | 1 | ✅ |
| 職務與人才規格 14 節 | 14 | ✅ |
| 成熟度彙整 | 1 | ✅ |
| 職務招募說明 | 1 | ✅ |
| **小計（使用者主動按）** | **23** | ✅ |
| 整合層（每次整理後自動跑，有 400 字門檻節流） | ≤22 | ❌ 不計入 |

每次呼叫約 1,700–4,000 input tokens ＋ 約 900 output tokens。

**配額設 30 次 = 完整跑完一輪（23 次）＋ 7 次重做的餘裕。** 需要取捨，取捨才會思考——這是刻意的。

> 成本注意：整合層不計入使用者配額，但**費用仍由平台承擔**，實際帳單約為配額次數的兩倍。若要進一步降低，可把整合層從「每次整理後」改為「跨階段時」觸發。

## 3. 資料模型

### 3.1 `jd_codes/{code}` — 課程授權碼

```js
{
  code:        'ORMAP-2607-A31F',   // 文件 ID 即為碼
  cohort:      '2026 秋季職務設計班',
  quotaTotal:  30,
  isInstructor:false,               // 講師碼：不扣配額
  redeemedBy:  null,                // 綁定的 uid，null 表示未使用
  redeemedAt:  null,
  revoked:     false,               // 停權用
  createdAt:   Timestamp,
}
```

一人一碼：`redeemedBy` 一旦寫入就不可更改，別人拿到同一組也無法使用。

### 3.2 `jd_users/{uid}` — 使用者的授權與用量

```js
{
  code:       'ORMAP-2607-A31F',
  cohort:     '2026 秋季職務設計班',
  quotaTotal: 30,
  quotaUsed:  7,                    // 只有 Cloud Function 能寫
  isInstructor:false,
  redeemedAt: Timestamp,
}
```

### 3.3 `jd_cases/{caseId}` — 案例（取代 localStorage）

```js
{
  ownerUid:   'uid...',
  case_id, path_type, fields, modules, case_summary,
  doc_status, doc_overrides,        // 沿用現有結構，欄位 ID 不變
  createdAt, updatedAt,
}
```

**永久保存**：不設自動刪除。畫面固定顯示提醒——「網站會持續改版，重要文件請自行下載保存」。

### 3.4 `jd_ai_logs/{logId}` — 用量稽核（僅後端可寫）

```js
{ uid, caseId, taskCode, inputTokens, outputTokens, model, at, counted }
```

用於對帳與異常偵測。使用者讀不到。

## 4. Firestore Rules

```js
match /jd_codes/{code} {
  // 碼不可被列舉；兌換一律走 Cloud Function
  allow read, write: if false;
}

match /jd_users/{uid} {
  allow read:  if request.auth != null && request.auth.uid == uid;
  allow write: if false;            // 只有 Cloud Function 能寫，避免自己改配額
}

match /jd_cases/{caseId} {
  allow read, update, delete: if request.auth != null
                              && resource.data.ownerUid == request.auth.uid;
  allow create: if request.auth != null
                && request.resource.data.ownerUid == request.auth.uid;
}

match /jd_ai_logs/{logId} {
  allow read, write: if false;      // 後端專用
}
```

要點：`quotaUsed` 與授權碼**前端一律不可寫**，否則配額形同虛設。

## 5. Cloud Functions 介面

### 5.1 `redeemCode(code)` — callable

驗證後把碼綁到目前登入的 uid，建立 `jd_users/{uid}`。

- 需登入
- 交易內檢查：碼存在、未被兌換、未停權
- 已兌換過其他碼的使用者不可重複兌換
- 回傳 `{ ok, cohort, quotaTotal, quotaUsed }`
- 錯誤碼：`NOT_FOUND` / `ALREADY_REDEEMED` / `REVOKED` / `ALREADY_HAS_CODE`

### 5.2 `jdAi({ taskCode, system, user, caseId })` — callable

取代目前的本機代理 `/api/ai`。

執行順序（**扣配額必須在呼叫供應商之前**）：

1. 檢查登入
2. 讀 `jd_users/{uid}`，沒有就回 `NO_ACCESS`
3. `taskCode === 'SYNTH-01'` 或 `isInstructor` → 不扣配額
4. 否則在**交易**內檢查 `quotaUsed < quotaTotal`，不足回 `QUOTA_EXHAUSTED`；足夠則 `quotaUsed += 1`
5. 從 Secret Manager 取 Key，呼叫供應商
6. 寫入 `jd_ai_logs`
7. 回傳 `{ ok, data, quotaUsed, quotaTotal }`

**供應商呼叫失敗時退還配額**（在交易外補回，並記錄）。使用者不該為系統錯誤付費。

回傳形狀與現有 `jd-ai.js` 完全一致，前端只需換 endpoint。

### 5.3 `issueCodes({ cohort, count, quota, isInstructor })` — callable，限管理者

批次產碼。僅 `counselors/{uid}.isSuperAdmin` 可呼叫。

## 6. 前端調整

### 6.1 `jd-ai.js`

`defaultProxy()` 改為兩種模式：本機代理（開發用）與 Cloud Function（線上）。`askAi()` 的回傳形狀不變，額外帶回 `quotaUsed / quotaTotal`。

### 6.2 送出前的品質檢核（新增，只提醒不擋）

按下「請 AI 幫我整理」前顯示，讓人自己判斷值不值得花這一次：

```
這次要送出 42 字。剩 8 次。

⚠ 內容和上次整理時幾乎一樣，再按一次結果會差不多
⚠ 只有形容詞（細心、積極），沒有具體情境
✓ 有提到實際發生的事
```

檢核項目：

| 檢核 | 觸發條件 | 提示 |
|---|---|---|
| 內容未變更 | 與上次送出的內容相似度過高 | 「再按一次結果會差不多，要不要先補一點內容？」 |
| 內容過短 | 少於約 60 字 | 「這段只有 N 個字，整理出來大概也是這個長度。要不要先多聊幾句？」 |
| 只有抽象形容詞 | 命中「細心／積極／負責／抗壓」等且無情境動詞 | 「這幾個詞可以再問問看：什麼情況下、會做出什麼行為？」 |
| 仍含引導者問句 | 命中問句樣式 | 「這裡面好像還有提問的句子，那不是回答。」 |
| 沒有具體事例 | 無時間、數量、對象等具體指涉 | 「有沒有一個實際發生過的例子？整理起來會準很多。」 |

全部只提醒，按鈕照樣可按——照既有原則，**審核是人**。

### 6.3 常駐顯示剩餘次數

工作區頂列固定顯示 `AI 次數 23/30`。低於 5 次時變色提示。

### 6.4 資料層 `jd-store.js`

介面（`listCases` / `getCase` / `saveField` / …）不變，實作從 localStorage 換成 Firestore。首次登入時把本機既有案例上傳，成功後保留本機備份。

### 6.5 保存提醒

案例列表與產出文件頁固定顯示：

> 網站會持續改版，重要文件請下載保存。

## 7. 前置條件（**必須先完成**）

授權與配額都建立在「這個 uid 是誰、是否有效」之上。缺陷登記簿中下列項目未修復前，不應開放付費或課程授權：

| 編號 | 問題 | 影響 |
|---|---|---|
| `AUTH-001` | `isActive`／管理者有效性前後端判定不一致 | 權限判定本身不可靠 |
| `SEC-001` | 評測草稿讀取範圍過寬 | 共用 Auth 骨幹 |
| `SEC-003` | 評測結果建立未要求登入 | 共用 Auth 骨幹 |
| `SEC-004` | 評測清單對所有已登入者開放 | 共用 Auth 骨幹 |

`SEC-005` 白板提權路徑雖已因功能移除而解除，但**殘留資料尚未清查**，一併處理。
