## 1. 結論摘要

系統目前**沒有任何「已作答」狀態**。`scores` 與 `currentScores` 在載入時被強制補 0
（`tools/twa.html:1269-1270`），此後全站一律以 `> 0` 當作「使用者有推進過」的代理訊號。
因此「刻意選 0」與「尚未作答」在資料層**完全無法區分**，且此資訊在寫入草稿與
`assessments` 之前就已經遺失 —— 不是渲染問題，是資料模型問題。

盤點中另發現一個比 0/0 更立即的問題，見 §3.3：**只填了現況、理想仍為 0 的需求，
報告會把它分類為「盈餘」**，而不是「未完成」。

## 3. 三種組合的實際行為

### 3.1 `0 / 0`（現況 0、理想 0）

| 環節 | 行為 |
|---|---|
| 評測畫面 | 與完全沒碰過一模一樣：階段 2／3 鎖定、目錄「未開始」、不計入進度 |
| 草稿（localStorage `LPK_TWA_V2026_ULTIMATE_FINAL`／`user_progress/{uid}_TWA`） | 仍會存入 `0`，但與未作答的 `0` 無法區分 |
| 儲存至 `assessments` | `rawScores`／`rawCurrentScores` 皆為 `0` |
| 報告 | `report.html:229` 回傳 `null` → **未評分**，不出現在三態與九宮格 |
| 雷達圖 | `twa.html:2396-2397` 直接把 0 納入類別平均 → **會拉低該類別百分比** |

⚠️ 注意最後一列：0/0 在三態被當成「沒作答」排除，卻在雷達圖被當成「真的 0 分」計入。
**同一份報告內對 0 的語意就不一致。**

### 3.2 `0 現況 / 正數理想`

正常路徑做不到（階段 2 需 `cur > 0` 才解鎖），但把現況滑桿拉回 0 即可產生。
報告會判為「渴望」，符合直覺，無立即問題。

### 3.3 `正數現況 / 0 理想` — 🚨 最需要注意

這是「只填了現況、還沒填理想」的**未完成中間狀態**，卻不會被 `report.html:229` 排除
（因為 `current !== 0`），於是進入三態分類：

- 現況 ≥ 2、理想 0 → `current - ideal >= 2` → **判定為「盈餘」**
- 現況 1、理想 0 → **判定為「平衡」**

也就是說，**使用者只要拉了現況分數就中途送出，該項會被報告說成「你現在擁有很多、但其實不在乎」**
（`report.html:293` 的 high-low 文案），而事實只是他還沒填理想值。

系統沒有任何送出前的完成度檢查（`twa.html:2485` 之後直接儲存），所以此情境**隨時可能發生**。

> 判斷：這一項與 0 分語意相關但**可獨立處理**，且不需要新資料欄位 —— 屬於分類函式的
> 邊界條件錯誤。建議優先度高於 0/0 本身。

## 4. 是否有既有欄位可沿用

**沒有。** 全域搜尋 `touched`／`answered`／`isAnswered`／`visitedNeeds` 於
`tools/twa.html`、`report.html`、`dashboard.html`、`timeline.html`、`admin.html` 皆零命中。

`assessments` 寫入的欄位（`twa.html:2497-2504`）為：
`name`、`email`、`job`、`goal`、`rawScores`、`rawCurrentScores`、`rawSelected`、
`rawNotes`、`radarScores`、`insight`、`top5`、`focusType`。
其中沒有任何完成度或作答狀態旗標。

---

## 5. 舊資料相容性

因為 `twa.html:1269-1270` 對所有 20 項強制補 0，**既有紀錄的每一項都必然有值**，
無法事後回推當初是刻意選 0 還是沒作答。

因此任何方案都必須遵守：**新欄位缺席時，一律沿用現行行為**。
不得對舊報告重新分類，否則同一份報告在不同時間會顯示不同結論。

## 6. 兩個方案

### 方案 A：加一組 `touched` 集合（最小變更）

**做法**：新增 `let touched = {}`。任一滑桿第一次 `input` 就記 `touched[id]=true`（**包含拉到 0 或停在 0**）。
把 §2.1 中「判定是否已作答」的位置改讀 `touched`，**保留** BUDGET_MAP、top5 等「用分數大小」的位置不動。

| 面向 | 內容 |
|---|---|
| 影響檔案 | `tools/twa.html`（狀態、渲染、草稿讀寫、儲存 payload）、`report.html`（分類 fallback） |
| 資料變更 | 草稿與 `assessments` 各多一個 `touchedNeeds`（id 陣列） |
| 舊資料 | `touchedNeeds` 不存在 → `report.html` 走現行 `>0` 判定，舊報告零變化 |
| 優點 | 資料形狀相容、可漸進、Rules 不需改、回復容易（刪欄位即回到現狀） |
| 限制 | 「碰過滑桿」不完全等於「想清楚後選 0」；仍可能誤記 |
| 回復方式 | 移除欄位寫入即可；舊有 fallback 保證不壞 |

### 方案 B：分數改為可為 `null`（語意最完整）

**做法**：取消 1269-1270 的補 0，未作答維持 `null`／`undefined`，選 0 就真的是 `0`。
`0` 與「未作答」在型別上就分開。

| 面向 | 內容 |
|---|---|
| 影響檔案 | `tools/twa.html` 全部 `|| 0` 慣用寫法（§2.1 幾乎每一列）、`report.html`、雷達圖平均、草稿還原、mirror 同步 |
| 資料變更 | `rawScores` 值域由 `number` 變 `number\|null` —— **資料契約變更** |
| 舊資料 | 舊資料全是 0，讀取端無法分辨，仍需 fallback；等於 B 也躲不掉 A 的相容層 |
| 優點 | 語意最乾淨，雷達圖可正確排除未作答項 |
| 限制 | 回歸面極大；`|| 0` 到處都是，漏改一處就靜默變成 0；風險與收益不成比例 |
| 回復方式 | 需整批回退，不易局部關閉 |

**建議**：採方案 A。方案 B 的額外語意收益，主要只體現在雷達圖平均值，
但要付出全檔 `|| 0` 改寫的回歸成本，且仍需要同一套舊資料 fallback。

> **實作補註（2026-07-22）**：已採方案 A。雷達圖**刻意維持原樣**（0 分照常計入類別平均），
> 理由有二：(1) 對「刻意選 0」而言，計為 0 本來就是正確的；(2) `radarScores` 於送出時
> 一併寫入 `assessments`，改動分母會讓新舊紀錄無法互相比較，影響 timeline 的跨期對照。
> §3.1 指出的語意不一致因此仍然存在，但已縮小為「未作答項也被計入平均」這一種情形。
