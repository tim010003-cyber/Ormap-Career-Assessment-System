## 1. 使用規則

### 1.1 何時必須登記

- 程式碼或 Rules 已顯示存取範圍大於產品預期。
- 現有功能在可重現條件下失敗、資料不一致或流程中斷。
- 前端、後端、Rules、索引或部署設定使用不同判定語意。
- 正式環境可能混入 Demo、mockup、測試資料或未核准內容。
- 缺少版本控制、預覽、回復或驗證證據，可能造成環境不一致。

只有分類而沒有具體紀錄，不算完成登記。

### 1.2 狀態

| 狀態 | 定義 |
|---|---|
| 待確認 | 有線索，但尚未取得足夠證據或預期行為尚未確認 |
| 已確認／未修復 | 已由程式碼或測試確認，尚未完成修復 |
| 修復中 | 已有核准工作包與唯一寫入者 |
| 待驗證 | 已實作修正，但尚未完成必要回歸或部署驗證 |
| 已驗證 | 驗收案例通過；若涉及正式環境，已核對部署版本 |
| 已接受風險 | 使用者在理解影響後明確接受；需記錄期限或重評條件 |
| 不處理 | 經使用者決策不處理；需記錄原因與影響 |

### 1.3 嚴重度

| 等級 | 定義 |
|---|---|
| P0 緊急 | 已知正在造成重大資料外洩、不可逆資料破壞或全面不可用 |
| P1 高 | 可能造成未授權存取、資料篡改、核心權限失效或重大正式流程回歸 |
| P2 中 | 可能造成環境不一致、部分流程失敗、營運風險或明顯信任落差 |
| P3 低 | 局部顯示、命名、文案或非核心體驗問題 |

嚴重度是目前判斷，不是事實本身；取得新證據後可以調整，但必須留下變更原因。

### SEC-001｜評測草稿讀取範圍過寬

- **狀態**：已確認／未修復。
- **嚴重度**：P1 高。
- **確認依據**：`firestore.rules:53-56`；`user_progress/{docId}` 的寫入以目前使用者 UID 前綴限制，但讀取只要求已登入。
- **受影響資料／功能**：TWA 雲端草稿、自動儲存與恢復；目前 HTI 主要使用瀏覽器 localStorage，是否另有雲端草稿相依尚未驗證。

- **期望行為**：一般使用者只能讀寫自己的草稿；如需管理或諮詢師支援，另以明確角色與資料關係授權。

- **驗收條件**：本人可建立、讀取、更新與刪除自己的草稿；其他一般使用者讀取或列出時被 Rules 拒絕；TWA 自動儲存與恢復回歸通過；Rules 已部署至目標環境並完成拒絕案例驗證。

### AUTH-001｜`isActive`／管理者有效性判定不一致

- **狀態**：已確認／未修復。
- **嚴重度**：P1 高。
- **確認依據**：
  - `firestore.rules:5-12`：`isCounselor()` 將 `isActive` 未設為 `false` 視為有效；`isSuperAdmin()` 只檢查 `isSuperAdmin == true`。
  - `auth-utils.js:38-61`：前端 `isCounselor()` 與 `isSuperAdmin()` 都要求 `isActive === true`。
  - `counselors.html` 另有 `isActive !== false` 的顯示與統計判定。
- **受影響資料／功能**：諮詢師、Super Admin、報告存取、後台與 Firestore Rules。
- **可能影響**：同一帳號可能在畫面上被視為停用或無權限，但在 Rules 層仍符合部分權限；也可能造成管理畫面與實際資料權限不一致。
- **期望行為**：單一、明確、fail-closed 的有效帳號契約；前端、Rules、管理頁與文件使用同一語意，並明確處理 `isDeleted`。
- **修復工作包**：尚未建立。
- **驗收條件**：針對欄位為 `true`、`false`、缺值，以及已刪除／非管理者組合建立權限矩陣；前端與 Rules 結果一致；停用與刪除帳號無法保留舊權限；既有有效管理者仍可操作。
- **尚待確認**：舊資料中缺少 `isActive` 或 `isDeleted` 的文件數量與遷移策略；不得為確認數量而輸出真實人員資料。

### SEC-005｜白板結束流程可用於取得他人評測報告存取權

- **狀態**：已確認／未修復。**嚴重度**：P0 緊急。**發現者**：Claude（INTF-001 核對過程）。
- **確認依據**：`firestore.rules:63-65`（session read＋write 只要求已登入）＋`whiteboard.html:320`（角色判定為純客戶端）＋`whiteboard.html:552-573`（`confirmEnd()` 依 `studentUid` 批次寫入 `assignedCounselorUid` 與 `counselorAccess:true`）＋`firestore.rules:42-45`（`assessments` update 只要求 `isCounselor()`，未驗證關係）。
- **受影響資料／功能**：`assessments` 的授權欄位、全部 TWA／HTI 報告的可讀範圍。
- **實際行為**：session 的 `studentUid` 可被任意登入者改寫，而結束流程完全信任該欄位決定綁定對象。
- **期望行為**：`assignedCounselorUid` 的寫入須由伺服端約束——寫入者為合法諮詢師，且存在 `counselorUid == 寫入者 && studentUid == 目標` 的 session。

- **驗收條件**：非參與者的 session 欄位寫入被拒；`assessments.assignedCounselorUid` 越權寫入被拒；既有資料完成盤查；正常結束流程回歸通過。
- **尚待確認**：既有紀錄中非預期綁定的數量（需使用者授權方可查詢正式資料）。

- **確認依據**：`whiteboard.html:623-631`（`getDoc` 失敗時 catch 為空，回退至 URL 參數 `?c=`／`?t=`）＋`whiteboard.html:646-711`（樣板字串直接插入，未跳脫，同時進入屬性語境 654 與文字語境 655）。同類：`report.html:509`（HTI 分支未跳脫 `userName`，而 TWA 分支 `report.html:812` 有 `escHtml`，兩分支不對稱）、`timeline.html:217-229`（`focusType` 直接內插）。

### SEC-007｜HTI 草稿未綁定 uid

- **狀態**：已確認／未修復。**嚴重度**：P1 高。
- **確認依據**：`tools/hti.html:665` 以固定 key `hti_progress_v4` 寫入 localStorage，不含 uid；`tools/hti.html:675-677` 無條件提示還原。對照 TWA 走 `user_progress/{uid}_TWA`（`tools/twa.html:2692`）具天然隔離。
- **實際行為**：共用裝置上後登入者會被提示還原前一位使用者的作答；若接受並存檔，該作答以**後登入者的 uid 與 email** 寫入 `assessments`（`tools/hti.html:1327-1339`）。
- **期望行為**：localStorage key 綁定 uid，或還原前比對 uid。
- **驗收條件**：不同帳號在同一瀏覽器不互見草稿；既有草稿的遷移或捨棄行為明確；HTI 作答、還原、存檔回歸通過。
- **註**：純前端修復，不需改 Rules，可與 Rules 工作並行。

### OPS-002｜白板 HTI 同步寫入頻率過高

- **狀態**：已確認／影響未量化。**嚴重度**：P2 中。
- **確認依據**：`tools/hti.html:1252` 的 `setInterval(_wbDoSync, 3000)` 無條件寫入、不比對差異，與 `tools/hti.html:1174` 的 500ms debounce 重疊；僅在 session 結束時清除（1258），分頁隱藏或導覽離開不停止。TWA 無此輪詢（僅 debounce）。
- **可能影響**：單場 60 分鐘會談約 1200 次寫入。成本與配額風險，非安全風險。
- **期望行為**：改為變更驅動，與 TWA 一致。
- **驗收條件**：鏡像即時性未退化；寫入次數顯著下降；session 結束與離開頁面皆正確停止。

### ⚠️ SEC-005 的殘留資料（功能移除**不等於**風險清零）

白板功能已移除，攻擊路徑不再存在。但 SEC-005 的授權是**寫進資料**的：
`assessments` 文件中的 `assignedCounselorUid` 與 `counselorAccess: true` 一旦被寫入就會留存。

- 白板移除後，**沒有人能再新增**這類授權。
- 但**既有的授權仍然有效** —— `report.html:340-344` 與 `firestore.rules:42-45` 仍會依這兩個欄位放行。
- 因此：**尚未清查前，不得宣稱 SEC-005 已完全解除。**

待辦：盤查 `assessments` 中的 `assignedCounselorUid`，確認每一筆都對應真實的諮詢關係；
不符者清除該欄位與 `counselorAccess`。需使用者授權後執行（見 SEC-PLAN-001 §6A）。
