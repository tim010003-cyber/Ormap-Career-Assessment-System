## 一、確立的原則

**AI 是提醒與整理，不是審核者。審核是人。**

使用者的原話：

> 不論準不準、好不好，這件事情由我來決定，不是 AI 來決定。AI 都是作為整理用的，所以它就算產出一個非常有偏見的東西也沒關係，我會帶著它去討論、整理出來。
>
> 我們在討論和撰寫這件事情時，它是充滿彈性的，不可能交由 AI 去做審判。很多事情是預測不到的，所以我們一定是當下去做討論與評估。

推論出來的具體規則：

1. AI 不判定「不完整」「不合格」「不一致」「有矛盾」「不可發布」。
2. AI 不因為輸入短、模糊或偏頗而拒絕整理。
3. AI 產出有偏差沒關係——人會帶著草稿到現場討論。**不必為了正確而留白。**
4. AI 可以「提醒」，不可以「裁決」。提醒是「這裡我看到兩種說法」；裁決是「這裡有矛盾，必須先解決」。
5. 結構化欄位是**收納格，不是待辦清單**。空著是正常的。
6. Step 1（問題釐清、職務需求界定）現場是一題一題來回問出來的，AI 每次拿到的只是片段。**片段就整理片段**，不要假設人應該一次講完。

## 二、已移除的「AI 審核」清單

| 原本 | 誰在審核 | 改成 |
|---|---|---|
| `cannot_complete`（空白輸入） | AI 判定輸入不合格 | 回空整理 +「想先跳過、之後再回來補都可以」 |
| 兩處 `alert()` 打斷 | 系統攔截操作 | 移除，不再打斷 |
| QA-02 關鍵字黑名單擋發布 | AI 判定不可發布 | 發布前自我複查清單，`pass` 恆為 true |
| QA-01 一致性檢查 | AI 判定不一致 | 對照清單，用「想留白也沒關係」的語氣 |
| `moduleUnlocked()` 階段硬鎖 | 系統決定順序 | 恆為 true，可自由跳轉 |
| M1「尚未辨識出已確認事實，多為感受或推測」 | AI 判定回答品質 | 刪除。這階段本來就常只有感受 |
| M1「N 項問題未能歸類，請人工判斷」 | 把 AI 分類失敗說成使用者的缺漏 | 「有 N 項我沒把握該歸到哪一類，你來看比較準」 |
| M3「預期價值少於三項，正式文件要求三至六項」 | 把參考範圍當門檻 | 刪除 |
| `bucketize` 靜默吞噬未命中內容 | AI 悄悄決定什麼重要 | 記錄到 `_unmatched`，可誠實說「這幾句我不確定該放哪」 |

### QA-02 的具體差別

改前（誤判高、且會擋）：

```js
if (/L[1-5]\b/.test(t))                  // 「L3 產線」也中
if (/最低可接受等級|評分|計分|分數/…)     // 「分數」是正常用詞
organized_content: { pass: hits.length === 0 }   // 命中就不給發布
```

改後：只在等級語境（前後有「等級」「level」「達到」「至少」）才視為內部代碼；`pass` 恆為 true；文案從「揭露檢查未通過」改成「發布前，這幾個地方你自己看一眼」，每則都用問句結尾把決定權交回去。

已實測：
- `「L3 產線待過，熟悉分數統計」` → **0 則提醒**（原本會被擋）
- `「需達到等級 L4，行為錨點見附件，結構化追問請參考評分表」` → **3 則提醒，但仍可發布**

## 三、`jd-transform-rules.js` 的 STANCE

每次呼叫 AI 都會送出，開頭第一句就是角色定位：

```
【你的角色：整理者，不是審核者】
審核由人做，不是由你做。你的產出是草稿，會被帶到現場由人討論、修改、推翻。
```

M1／M2 另外附掛一段：

```
【這一段不要架構化】
上面的欄位是收納參考，不是待辦清單。哪個格子有東西就放，沒有就空著，
不要為了填滿而追問或補寫。回應者只講了一兩句也照樣整理。
```

## 四、沒有動的部分

真正的專業判準全部保留：不得把「缺人」當已證實問題、不得把時間先後當因果、不得把流程問題轉嫁成人才條件、不得用年資學歷代替可觀察行為、不得把所有職能提到 L5、不得把表現高於需求判為不適任、不得在對外 JD 公開錨點與評分。

這些約束的是 **AI 的行為**，不是使用者的行為。它們不會擋住任何人，只會讓 AI 整理得更專業。
