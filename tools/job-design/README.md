# 職務設計與甄選 Web App — 前端原型（Phase 2 路徑 B 垂直切片）

> 狀態：**可 demo 前端原型**，純前端 + 本地 Mock，**未接付費後端**。
> 對應規格：課程資料夾 `AI App規格/00–08`；技術方案見 `PRD/職務設計與甄選App_技術盤點與開發計畫_v1_2026-07-18.md`。

## 這是什麼

課程「面試的技巧與心法」搭配的 Web App。學員口述既有職務 → AI 整理與檢查 → 逐項人工確認 → 產出職務說明書、甄選流程與 JD → 合併成一份 Word。**AI 只整理與提問；要不要建立、最低標準、發布內容，都由人決定。**

本原型完整走通**路徑 B（既有職務優化）**：
建立案例 → 既有職務基礎確認 → PATH-01 完整度檢查 → M3 職務與人才規格（口述 → Mock AI 整理 → 四欄確認 → 五級行為錨點）→ M4 職務說明書 → M5 甄選流程 → M6 JD（+ QA-02 揭露檢查）→ 合併 Word 下載 → 72 小時關閉倒數 → 講師重開。

## 檔案

| 檔案 | 角色 |
|---|---|
| `index.html` | 案例列表、建立案例、講師／學員身分切換 |
| `case.html` | 案例工作區：模組步進器 + 招牌「原始→AI整理→AI推論/缺口→人工確認」UX |
| `jd-fields.js` | 欄位字典與模組設定（沿用規格 field_id，資料驅動） |
| `jd-store.js` | 持久層（localStorage）+ 狀態機 + 存取窗/72h 關閉 |
| `jd-mock-ai.js` | Mock AI Provider（PATH-01、M3-01~09、M4-01、M5-01~03、M6-01、QA-02），照 04 輸出契約 |
| `jd-word.js` | 產生**四份獨立文件**、下載 `.doc`、待確認彙整 |
| `jd-seed.js` | 導覽模式的預建示範案例（①完整流程、②權責矛盾） |

## 輸出：四份獨立文件（PO 決策 2026-07-19）

取代 `05_Word輸出規格` 原本的「單一合併 Word」。四份各自有封面：

| 檔案 | 內容 | 揭露 |
|---|---|---|
| ① 引導與思考 | 思考歷程（第一部基礎確認＋第二部職務與人才規格）＋**待確認事項**＋版本摘要 | 內部 |
| ② 職務說明書 | 正式輸出，**不含思考引導內容** | 內部 |
| ③ 甄選流程設計書 | 流程、面試題、行為定錨、培訓缺口 | 內部 |
| ④ JD 對外文案 | 唯一可對外的檔案 | **可對外** |

揭露邊界改由**檔案層級**界定（封面標示），比原本單一合併檔的章節註記更安全。

## 本地預覽

ES modules 需經 HTTP（`file://` 會被 CORS 擋）。用任一靜態伺服器指向 repo 根目錄，例如：

```
npx serve .        # 或 firebase serve
```

然後開 `http://localhost:<port>/tools/job-design/index.html`。

## Demo 取捨（與正式版差異）

| 項目 | 原型 | 正式版（v1 計畫） |
|---|---|---|
| 身分 | 畫面切換講師／學員 | Firebase Auth Google 登入 + `jd_users` 角色 |
| 資料 | localStorage | Firestore（`jd_*` 集合）+ 逐筆權限 Rules |
| AI | 前端規則式 Mock | Cloud Functions callable + Secret Manager + 真實供應商 |
| Word | 前端 HTML-based `.doc` | Cloud Function `docx` → Storage 簽名網址 |
| 72h 關閉 | 本機時間 + 頁面倒數 | Cloud Scheduler `onSchedule` 冪等排程 |

**欄位 ID、模組流程、確認節點、資料形狀刻意與正式版一致**，日後換後端不需改資料模型。

## 目前收錄的欄位範圍

各模組收錄「足以展示完整機制」的代表性欄位（非全 150 欄）。欄位表資料驅動，補齊其餘欄位只需在 `jd-fields.js` 加設定。
