## 一、結論摘要

目前體驗不佳不是單一提示詞問題，而是三層問題疊加：

1. **上游口語沒有先正規化**：部分 Mock 邏輯直接把口語片段當成能力、工具或判斷項目，後續句型只能把未整理原料繼續放大。
2. **後半流程仍由固定模板主導**：M3 成熟度與 M5 甄選流程／結構化面試不會呼叫真 AI；即使真 AI 已啟用，這兩段仍會產生泛用、重複、僵化的內容。
3. **產品一次要求過多正式產出**：還在探索職務時，就要求完整五級成熟度、每項職能兩道題、七欄面試表與四階段流程，負擔大，也把初步口述過早包裝成看似正式的規格。

建議不要只加強 `COMMON_RULES`。新版應先建立「口語內容 → 精煉事實／決策／工作／能力候選」的獨立整理層，再讓使用者選擇是否進入成熟度與甄選設計。結構化面試建議改成**選配的第二階段產物**，只針對人工選出的少數核心能力生成。

## 三、AI 後臺目前實際怎麼運作

### 3.1 路徑總覽

| 階段 | 前端動作 | 真 AI | Mock | 說明 |
|---|---|---:|---:|---|
| PATH-01 | 既有資料完整度整理 | 否 | 是 | 規則式檢查 |
| M1 問題釐清 | 口述逐章整理 | 是 | 是 | 先產 Mock，再視本機代理狀態嘗試真 AI |
| M2 需求界定 | 勾選、原因、後續行動彙整 | 否 | 是 | 規則式矛盾與提醒 |
| M3 各節 | 口述轉職務／能力欄位 | 是 | 是 | 真 AI 可用時覆蓋 Mock organized content |
| M3 成熟度 | 五維度 L1–L5 | 否 | 是 | 固定通用五級加同一段情境註記 |
| M4 職務說明書 | 組裝欄位 | 否 | 是 | 主要是既有欄位組裝 |
| M5 甄選流程／面試題 | 四階段流程、每項能力兩題 | 否 | 是 | 完全由固定模板產生 |
| M6 對外 JD | 候選人版文案 | 是 | 是 | 先產 Mock，再嘗試真 AI |
| QA-01／QA-02 | 對照與揭露提醒 | 否 | 是 | 規則式提醒，不應代替人工決策 |

### 3.2 真 AI 與 Mock 的切換

1. `case.html` 先執行 `jd-mock-ai.js`，得到一份保底 envelope。
2. `jd-ai.js` 查詢本機代理 `/api/status`；只有 `ok` 與 `hasKey` 同時成立才嘗試 `/api/ai`。
3. 本機代理未啟動、未設定、逾時或失敗時，沿用 Mock；原始輸入不清除，流程可繼續。
4. 前端只保存代理網址，不取得供應商 API key。

這個設計在技術安全與離線可用性上合理，但有一個產品誤解風險：畫面上都叫「AI」，使用者不容易知道某段其實永遠是固定 Mock。尤其成熟度與結構化面試品質差時，容易被誤判為真 AI 整理能力差。

### 3.3 prompt、轉換規則、schema 與 envelope

- `COMMON_RULES` 已朝「顧問整理助手」撰寫，要求抓重點、合併重複、把口語改成精準專業文字，並保留不確定性。
- `transformRule(taskCode)` 追加各任務轉換規則。
- `shapeHint()` 以 Mock 的 `organized_content` 當輸出形狀樣本，要求真 AI 回傳相同欄位。
- envelope 主要欄位：
  - `organized_content`：寫回正式畫面欄位的草稿。
  - `inferences_requiring_confirmation`：AI 補的角度或推論。
  - `information_gaps`：尚未談到或可補充的資料。
  - `contradictions`：並列的不同說法。
  - `warnings`：延伸提醒或揭露檢視。
  - `cannot_complete`：未知任務或技術上無法形成結果。
- `mergeIntoEnvelope()` 會讓真 AI 的 `organized_content` 覆蓋同名 Mock 欄位。陣列欄位只要真 AI 有明確回傳，即使是空陣列也以真 AI 為準；只有欄位缺失才回退 Mock，因此真 AI 的空 warning 不會被 Mock warning 污染。

### 3.4 回傳如何進入畫面與流程

- M1、M3、M6 由 `runWithAi()` 執行切換，再以 field ID 寫回 localStorage 案例資料。
- AI 整理與 AI 推論都可被人工修改；M1／M3 畫面會顯示來源、提醒與接受按鈕。
- 工作區目前已出現非阻擋式調整：M1、M2、M3 的下一步可用，warnings 顯示為延伸思考，未逐項接受 AI 欄位也可繼續。
- M5 仍使用「確認甄選流程，進入 JD」語意，但其內容是固定模板即時產生，沒有先讓使用者選擇哪些能力值得出題。

### 3.5 歷史根因：為什麼早期版本特別嚴苛

早期機制盤點確認三個根因，這些內容是理解目前程式演變的重要背景：

1. **把「不得補造事實」誤解成「不得改寫形式」**。原規格禁止 AI 虛構數字、因果與未提供事實，但仍要求 AI 合併重複、重寫句子、分類與轉成情境化行為。早期 prompt 只強調不能補造，沒有積極告訴 AI 應如何整理，模型因此傾向少寫、留白或全部丟進待確認。
2. **逐節轉換規則沒有送進 prompt**。角色定位、主要工作、必要知識、專業判斷、工具、等級與成熟度原本各有不同轉換原則；早期每個任務卻只收到同一份全域禁止清單。後來新增 `jd-transform-rules.js`，並由 `transformRule(taskCode)` 把各節規則接在 `COMMON_RULES` 後送出。
3. **把「不得阻塞」實作成硬鎖**。早期空白輸入、未逐格確認、未設定最低等級、前階段未完成、M3 必要欄位未確認及 QA-02 命中，都可能阻擋下一步或發布，和原規格「能回答多少就整理多少、保留人工繼續」相反。

早期 QA-02 也存在方向錯置：它以關鍵字黑名單判斷「不可發布」，容易誤判一般文字，卻沒有真正比較「對外必備要求是否高於甄選最低可接受等級」。目前 QA-02 已改為不阻擋的人工複查提醒，但「內外門檻實質比較」仍不是完整的自動檢查。

### 3.6 已完成的嚴苛度鬆綁

下列是工作區目前已可觀察到的鬆綁結果；它們解決「被卡住」，但尚未解決後半段輸出品質：

| 早期行為 | 目前行為 |
|---|---|
| 空白或短輸入被判定無法完成 | 回傳空整理或目前片段，可略過、稍後補充 |
| 每個 AI 欄位都要逐格確認 | AI 欄位是可編輯草稿，未逐格接受仍可繼續 |
| 前一階段未完成就鎖住導覽 | M1、M2、M3 可繼續往下，不以 AI 品質作完成閘門 |
| information gaps／inferences 被當成缺陷 | 顯示為待補充或待確認推論，不阻擋流程 |
| warnings 具有審核語氣或發布阻擋效果 | 改為延伸思考／發布前自己看一眼 |
| QA-02 命中即不可發布 | `pass` 維持 true，最終發布判斷交回人 |
| 未分類內容可能被靜默忽略 | Mock 可保留未能歸類的片段，讓人決定 |

目前 prompt 也加入「整理者，不是審核者」的立場：欄位是收納參考，不是待辦清單；使用者只講一兩句時就整理這一兩句，不為了填滿而追問或補造。

### 3.7 必須釐清的一項新舊決策差異

舊交接文件曾保留「不得把缺人當已證實問題」的專業守則。這句話若套用到所有情境，會再次否定使用者已做出的增員決策，因此最新規則應拆成兩種情況：

- 使用者只說「最近很忙，可能缺人」：AI 不應擅自把「缺人」當成已證實根因，可列為假設或待討論角度。
- 使用者明確說「已核准增員／已決定找人／就是要補人」：這是**已確認的決策背景**，AI 應忠實保留，不得要求使用者重新證明是否需要人。

兩種情況都可以提醒工作範圍、成功條件、授權、資源與合作形式，但提醒不阻擋流程。這項最新區分取代舊文件中一律「不得把缺人當已證實問題」的寫法。

## 四、測試輸出的具體問題

### 4.1 口語沒有真正轉成精煉內容

**已確認現象：**

- 課程助教 JD 的角色定位與工作項目有基本結構，但工具、品質、倫理與成熟度等段落仍保留大量「我覺得、目前會使用到、大概就是」等口述痕跡。
- 對外能力句出現「你具備……相關知識」的機械套句；測試文件中「相關知識」出現 8 次，並有「法律相關的知識相關知識」等重複語意。
- 原始的 Google Sheet、LINE、英語能力等片段，沒有先被辨識為「工具使用」「溝通情境」「語言能力及使用門檻」，而是被同一個知識句型處理。

**程式原因：**

- 上游 `buildAnchors()` 可以直接使用口語片段作為 `competency`。
- M6 Mock 只以能力名稱中是否含「工具／系統／操作」「方法／技術／流程」等關鍵字選句型；未命中時一律退回「你具備 X 相關知識」。
- 因此，即使 M6 再用真 AI 改寫，輸入的能力分類已經污染，模型要同時猜測分類、去重、精煉與寫招募文案，品質不穩定。

### 4.2 成熟度五級是「同一段口語 × 五個通用模板」

`runMaturity()` 會取每個維度口述的第一段，組成 `依本職務情境：……`，再把完全相同的情境註記附在 L1–L5 五個通用描述後。這不是依工作情境產生五種可觀察行為，只是五段通用文字加同一段備註。

影響：

- 五級看起來很完整，實際辨識度低。
- 使用者尚未掌握細節時，必須一次處理五個維度、每個五級，認知負荷過高。
- 「五個維度都需要填寫」「挑三個」等設計仍帶有強制完成感，和「初步討論也可往下」的產品方向衝突。

### 4.3 結構化面試把所有錨點等量展開

`runM5()` 固定建立四個階段，然後對 `mergeAnchors()` 合併進來的每一個知識、工具、態度與成熟度錨點，各產生兩道相同結構的題目：一題過去經驗、一題資源更少的未來情境。

**已確認輸出：**

- 甄選流程文件有 6 張表、69 列、453 個 `th`／`td` 儲存格。
- 「成熟的工作者與不成熟的工作者的差異」被當成職能名稱並重複出題。
- 空白、句子型或過長的能力名稱也會直接嵌入「請描述一次你實際處理 X 的經驗」。
- 每個錨點兩題，沒有去重、核心能力排序、題數上限，也沒有判斷這個工作是否真的需要四個甄選階段。

這會讓「結構化」退化為「格式一致」，而不是針對相同能力取得可比較證據。

### 4.4 流程時間語意互相矛盾

測試 JD 中整體流程描述為 1 小時，但四個階段各自又套入 1 小時，共出現 5 次一小時。原因是 `candidate.duration` 與 `candidate.stage_time` 被分別套到總流程與每一個固定階段，沒有區分「總時長」和「單階段時長」。

### 4.5 長表格看不到完整文字

**已確認程式與文件結構：**

- M5 問題表固定 7 欄：階段、職能、題型、核心問題、追問、行為證據、等級。
- 畫面使用 `overflow-x-auto`，但儲存格字級小，沒有欄寬策略或重點／展開層次。
- HTML-based `.doc` 的共用 CSS 只有 `table { width:100% }` 與統一儲存格 padding／字級，沒有 `table-layout`、指定欄寬、橫向頁面或分卡策略。

這不是單純 CSS 小問題。當每個能力同時帶七欄長文字時，即使改成橫向，也會有閱讀密度過高的問題。

### 4.6 檔名與內容不一致

`JD 對外文案_資深客服專員_v1_20260721.doc` 的內容實際是「課程助教與活動專員」的職務說明書／表格內容，且找不到「資深客服專員」。

目前只能確認**檔名與內容不一致**；尚不能判定是下載時案例狀態切換、選錯文件、舊檔重新命名，或同步修改中的匯出問題。後續需以兩個案例連續下載做可重現測試，不能直接把根因歸給 `downloadDocument()`。

## 五、是否過度侷限正職員工

### 已有改善

工作區 `jd-fields.js` 已加入僱傭與非僱傭合作形式，包括全職、兼職、約聘、接案／外包、短期專案、顧問、實習等；「直屬主管」也開始改為「主要對接窗口」。這表示資料欄位層已有往多元合作型態移動。

### 仍有偏向

- M5 固定角色為「招募／HR、職務主管、團隊資深成員、單位主管」，更適合有完整 HR 流程的中大型組織。
- 固定四階段與正式行為定錨，對兼職助教、短期活動人力、接案者或小型團隊可能成本過高。
- 「到職」「不適任」「職稱／薪資」等語彙仍以僱傭關係為預設。
- 文件輸出把所有案例都推向完整 JD＋甄選設計，沒有「合作簡章／任務需求書／簡化甄選」等較輕量成果。

因此，問題不只在選項。後續需要讓合作形式真正影響流程深度、用語、文件名稱、甄選階段與必要欄位。

## 六、建議的新版產品順序

### 6.1 第一層：先完成可用的職務草稿

預設只要求：

1. 為什麼現在需要這個角色或合作。
2. 已確認決策（例如已核准增員、已決定找接案者）。
3. 希望交付的結果。
4. 主要工作與責任邊界。
5. 最低必要條件與可培訓項目。
6. 合作形式、投入量與主要對接者。

AI 的首要任務是把口語整理成短、準、可編輯的草稿，不產生五級，也不自動產生完整面試題庫。

### 6.2 第二層：人工選出少數核心能力

先把候選能力正規化成：

- 短名稱：例如「活動現場協調」。
- 一句定義：這項能力在本職務中代表什麼。
- 可觀察情境：在哪些工作場景看得見。
- 來源：使用者明說／AI 整理／AI 推論。

系統可建議 3–5 項，但使用者可刪除、合併、改名或略過。只有人工保留的核心能力才進下一層。

### 6.3 第三層：選配甄選工具

讓使用者選擇：

- 不需要：只輸出 JD／合作需求。
- 簡化：每個核心能力一題＋一個追問。
- 完整：結構化題目、行為證據、評分準則與工作樣本。

成熟度也改為選配：先做「最低可接受表現」與「理想表現」兩級，只有確有管理、培訓或升遷用途時才展開 L1–L5。

### 6.4 第四層：依合作形式調整深度

| 合作情境 | 建議預設 |
|---|---|
| 全職／長期核心職務 | 可選完整職務規格與甄選 |
| 兼職／活動支援 | 精簡工作結果、排班／投入量、現場情境題 |
| 接案／外包 | 交付物、驗收標準、溝通節點、權限與保密 |
| 顧問／諮詢 | 問題範圍、決策權、預期建議與投入上限 |
| 短期專案 | 里程碑、期間、資源、結案條件 |

## 七、後續建構建議

### P0：先修正輸入品質與產品順序

1. 新增獨立 `normalizeOralInput` 任務，不直接沿用原句作為 competency。
2. 要求輸出分為 `confirmed_decisions`、`facts`、`work_items`、`competency_candidates`、`open_items`，並保留來源文字對照。
3. 對能力候選做空值、過長句、重複、工具／知識／行為錯置檢查。
4. 把 M3 成熟度與 M5 從預設必經改為選配；不因略過而阻擋文件輸出。
5. UI 明示目前使用「AI 整理」或「本機模擬整理」，避免把 Mock 模板誤認成真 AI。

### P1：重做成熟度與結構化面試

1. 先由人工選 3–5 項核心能力，再生成題目。
2. 每項預設一題，追問依可觀察證據產生；不要無條件生成過去＋未來兩題。
3. 題目須使用具體工作情境，不直接把能力名稱塞進模板。
4. 甄選階段依組織規模、合作形式、風險與可用角色產生，不能固定四階段。
5. 時間 schema 分開：`total_duration`、`stage_duration`，避免總時間與各階段重複。
6. 成熟度先支援最低／理想兩級；完整 L1–L5 另行展開。

### P2：改善呈現與文件輸出

1. 畫面以能力卡呈現，預設只顯示核心題目；追問、證據、等級點開才顯示。
2. 文件改為「每項核心能力一個區塊」，避免七欄寬表。
3. 若保留表格，需定義欄寬、分頁、重複表頭、橫向頁面與長文字換行規則。
4. 為連續下載兩個案例增加 filename／case snapshot 一致性測試。

## 八、建議驗收案例

### 8.1 口語轉精煉

輸入：含重複、插話、模糊語助詞的長段口述。  
通過條件：輸出為 3–6 個不重複重點；不保留「我覺得、大概、就是」；不補造數字；每點可追溯來源。

### 8.2 已確認增員

輸入：「這個缺額已核准，我就是要找一位活動助教。」  
通過條件：列入 confirmed decision；可提醒範圍、成功條件、授權與資源，但不得再以「是否真的需要人」阻擋。

### 8.3 非正職合作

輸入：「這不是正職，我要找三個月的活動接案支援。」  
通過條件：不產生到職／直屬主管／完整 HR 四階段預設；輸出交付物、期間、投入量、對接與驗收。

### 8.4 資訊不足

輸入只有一至兩句初步想法。  
通過條件：仍產生暫定草稿，可人工修改、略過或繼續；缺口只標記待補充。

### 8.5 結構化面試

輸入含 12 個未整理能力候選。  
通過條件：先去重與要求人工挑選，不直接生成 24 道題；最終題目只對應選定核心能力，且每題有清楚證據目的。

### 8.6 文件可讀性與下載一致性

通過條件：A、B 兩案例連續下載時，檔名、封面職務名稱與內容一致；長題目在畫面與文件中可完整閱讀，不依賴極小字或大幅水平捲動。

## 九、安全邊界

本次建議放寬的是產品判斷與流程，不是技術安全。後續建構仍應保留：

- API key 不進前端、不寫入 localStorage、不出現在文件或版本控制。
- 不輸出、記錄或提交 token、cookie、secret。
- 不讀取 `.env` 或其他秘密檔內容。
- 不以測試名義呼叫付費 API；真 AI 驗證需另有明確授權與成本邊界。
- 不修改或部署 Firebase Auth、Firestore Rules、Hosting。
- AI 推論與使用者確認事實分開保存，不得把推論冒充事實。
- 對外 JD 的敏感資訊與內部面試內容仍需提醒人工檢視，但提醒不可成為自動發布裁決。
