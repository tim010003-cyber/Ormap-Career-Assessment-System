## A. 高嚴重度：結構性矛盾

### A1. 工作包要求驗證登入後行為，但同時禁止代理登入

**現象**
`DASHBOARD-TECH-REVIEW-007` 的交付項目包含：本人報告清單、空／錯誤狀態、查看報告、
再次探索（`?loadId=`）、刪除、授權開關、兩筆 TWA 軌跡、登入後手機／桌機回歸。
同一工作包又寫明「以真實登入或**核准的**預覽環境驗證」，而 `CLAUDE.md` 另有
「Browser、login、Emulator 或外部環境行為仍需適用的使用者授權與明確目標範圍」。

同樣情形也出現在 `ASSESSMENT-TECH-REVIEW-004`：草稿存取與還原、三階段作答、平手處理、
報告產出與儲存、`?loadId=`、完成後返回 Dashboard、`wb`／`mirror`，全部需要登入。

**衝突點**
交付清單裡**大多數項目在現行授權下永遠無法由代理完成**。代理只能回報「未測」，
於是每一次技術回歸的結論都必然帶著一大塊空白，而這塊空白剛好是風險最高的部分
（權限、資料寫入、刪除）。

**影響**
- 「可進預覽」的結論實質上只覆蓋未登入路徑，容易被誤讀為完整驗證。
- 使用者每次都要自己補測，但工作包沒有把這件事寫成正式交付步驟，容易漏掉。
- 累積下來，登入後的回歸從來沒有被系統性覆蓋過。

**可能選項（需使用者決定）**
1. 建立**測試帳號**並授權代理使用（需考量憑證處理方式；代理不得直接輸入密碼）。
2. 導入 **Firebase Emulator** 與種子資料，讓登入後流程可在本機重現，不碰正式資料。
3. 承認此界線，把「登入後驗證」正式改列為**使用者自測步驟**，並要求代理每次產出
   逐步自測清單（預期結果＋失敗時的證據要求），列為工作包的正式交付物。
4. 維持現狀（不建議：空白會持續累積且無人負責）。

### A2. 「完成」是否包含提交與部署，定義不明

**現象**
2026-07-21 實際發生：Codex 完成 `DASHBOARD-HOME-006`，在占用表登記
「**已完成並交還**／待 Claude 技術回歸與使用者產品驗收」，同時註明
「未提交、推送、合併或部署」。使用者依「已完成」的字面理解去看網站，
**看到的是舊版 Dashboard**，並回報「看起來沒什麼變化」。

同樣情形在同一天發生第二次：使用者查看公開網站改版時，開到的是上一輪的預覽網址與正式站，
兩者都不含當日成果。

**衝突點**
協議中「完成」目前指的是「檔案已寫入工作區」，但使用者的自然理解是「我看得到」。
中間隔著提交、推送、部署三個步驟，而協議**沒有指定這三步由誰負責、何時執行**。

**影響**
- 使用者無法驗收自己要求的功能，且會誤判代理沒做事。
- 成果長時間停留在未提交狀態，跨代理接手時容易被覆蓋（`WEB-IA-001` 已經發生過一次
  Dashboard 修改被另一分支寫入取代）。

**可能選項**
1. 定義三種狀態並強制在占用表標明：`已寫入工作區` / `已提交至分支` / `已部署至預覽`。
2. 規定每個工作包必須指定**發布執行者**，且「交還」時必須註明使用者可驗收的實際網址；
   若無網址則必須明寫「本工作包成果目前無法由使用者查看」。
3. 允許完成方直接提交到非 main 分支（不含部署），降低成果懸空的時間。

### A3. 未修復的 P0／P1 缺陷可否繼續發布，無規則

**現象**
缺陷登記簿目前仍有未修復的 P0／P1 項目（含 `SEC-005`、`SEC-002` 合併後的權限問題、
`AUTH-001` 權限語意、索引版本落差）。同時，專案持續產出預覽並準備合併到 `main`，
而合併 `main` 會**立即部署正式站**。

**衝突點**
協議寫了「已登記的問題不代表已修復」，但**沒有寫「未修復的 P0 是否構成發布阻擋」**。
目前每次發布判斷都由當次代理臨場決定，沒有一致標準。

**影響**
安全缺陷可能在無人明確承擔的情況下，隨著功能發布一起持續存在於正式環境。

**可能選項**
1. 明訂：存在未修復 P0 時，一律只能停在預覽，不得合併。
2. 明訂：P0 需使用者逐項簽署「知情並接受風險」才可發布。
3. 為每個 P0 設定期限與負責人，逾期自動升級為發布阻擋。

## B. 中嚴重度：規則缺口與不一致

### B1. `data/*.json` 手改與 sync 覆蓋的登記義務沒寫進協作規則

**現象**
`data/twa-landing.json`、`data/hti-landing.json` 是由 `data/*.xlsx` 經
`sync_landing_sheet.py` **整份重新產生**的檔案。`data/SYNC_GUIDE.md` 已兩次記錄
手改被覆蓋的教訓（第 95 行 `valueIntro.enabled`、第 124 行 `valueIntro.eyebrow`），
並要求手改後必須補「一次性事項」。

2026-07-21 的改名工作直接手改 JSON，**但未於 SYNC_GUIDE 補登記**。

**衝突點**
這條義務只寫在 `data/SYNC_GUIDE.md`，**沒有寫進 `AGENTS.md` 或 `CLAUDE.md` 的協作規則**。
代理若未被指示閱讀該指南，不會知道有這個義務。

**影響**
下次任何人執行 sync，改名可能**靜默還原**，無錯誤、無警示。且 SYNC_GUIDE 的登記
已與實際狀況不符（實際 3 筆、只記 2 筆），指南的可信度正在下降。

**補充：本次無法確認的部分**
執行審查的機器未安裝 Python／openpyxl，無法執行 `--dry-run` 判定覆蓋方向；
直接解壓 xlsx 亦未在 `sharedStrings.xml` 找到新舊任一字串。**因此覆蓋風險的實際嚴重度
目前是未知的**，這個不確定性本身應視為風險。

**可能選項**
1. 直接修改 xlsx 主檔使來源與 JSON 一致（最徹底，風險消失）。
2. 補一筆一次性事項到 SYNC_GUIDE（最低成本，但依賴下次有人閱讀）。
3. 把「`data/*.json` 為產生檔，手改必須同步登記」提升到 `AGENTS.md`／`CLAUDE.md`。
4. 在 sync 腳本加入差異警示，偵測到 JSON 與 xlsx 不一致時要求確認。

### B2. `firebase.json` 與 workflow 註解不一致，無「何者為準」明文

**現象**
`.github/workflows/firebase-hosting-pull-request.yml` 第 12 行註解寫明
ignore 清單「含 `mockups/**`、`tools/job-design/**`」。
但 `firebase.json` 的實際 ignore 清單**不含** `tools/job-design/**`。

已於預覽站實測確認：`tools/job-design/index.html` 回應 **200**，確實對外可存取。

**衝突點**
協議未規定「文件敘述與實際設定衝突時以何者為準」。本次審查採「實際設定為準」，
但這是代理的臨場判斷，非協議明文。

**影響**
執行合併的人可能依註解認為 job-design 不會上線，實際上會。
（本次已查明該目錄早於 `12c8eb0` 由使用者決策開放公開存取，故非新增曝光，
但註解本身仍是錯的，且會誤導未來的判斷。）

**可能選項**
1. 明訂「設定檔為唯一事實來源，註解僅供參考」，並修正該註解。
2. 若原意確實要排除 job-design，則修改 `firebase.json`（屬使用者決策，涉及公開範圍）。

### B3. 相似工作包對「可否修程式」給出不同規則

**現象**
- `ASSESSMENT-TECH-REVIEW-004`：「若為本次回歸造成，**claim 後可做最小修正**」。
- `DASHBOARD-TECH-REVIEW-007`：「**未另行 claim 前不得修程式**」。

兩者同為技術回歸工作包，措辭不同，實際授權範圍也不同。

**衝突點**
代理在發現缺陷時需要猜測本次適用哪一種。本次採較保守解讀（先回報、取得使用者授權後才修
`report.html`），但這靠的是判斷，不是規則。

**影響**
- 保守解讀 → 小問題卡住，來回一輪。
- 寬鬆解讀 → 代理在唯讀工作包中改了程式，違反使用者預期。

**可能選項**
1. 統一預設：技術回歸一律唯讀，發現缺陷只回報。
2. 統一預設：允許「與本次變更同範圍、且為單純錯字／文案不一致」的最小修正，其餘回報。
3. 在工作包模板中加入固定欄位「修正授權：無／最小修正／完整修正」，強制每次明寫。

## C. 其他待釐清

### C1. 產品名與概念用語的區隔未寫入命名規範

**使用者已於 2026-07-21 口頭定案**（建議正式寫入 `名詞對照表.md` 或品牌總綱）：

- **產品名稱／工具標題／報告標題**：「工作價值羅盤」（TWA）、「何倫興趣與行動風格」（HTI）
- **評測核心內容／概念用語**：「工作價值觀」「心理需求」「環境指標」「何倫特質」等
  **維持原寫法，不做替換**

理由：「工作價值觀」是心理學既有概念，本工具沿用並改造他人理論，須尊重原始定義。
原名「生涯價值覺察系統」因過於複雜而限縮；不採「工作價值系統」是因缺乏方向性。

**風險**：若無明文規則，未來的改名任務可能執行全域字串替換，把概念用語一起改掉。
本次審查已刻意保留 `data/twa-landing.json`、`tools/twa-intro.html`、`tools/twa.html`
中作為概念使用的「工作價值觀」，但這同樣是代理判斷，非規則。

### C2. `.gitattributes` 將 `*.html` 標為 binary，跨代理審查困難

**現象**
`.gitattributes` 標記 `*.html` 為 binary，`git diff` 預設不顯示逐行差異。
本次全程需以 `git diff --text` 繞過才能審查（`dashboard.html` 一次變更達 837 行）。

**影響**
- 跨代理交接時，接手方難以快速確認前一方改了什麼。
- GitHub PR 頁面上的 HTML 差異同樣不可讀，削弱 PR 預覽作為檢查點的價值。
- 本專案幾乎所有程式都在 `.html` 內，等於主要程式碼都不可 diff。

**可能選項**
1. 移除該標記，改以 `.gitattributes` 指定 `*.html diff=html`。
2. 保留標記，但在協作規則中明訂「審查 HTML 一律使用 `git diff --text`」。

---

### C3. 占用表「已交還／未占用」易被誤讀

**現象**
占用表已註明「『已交還／未占用』只表示目前沒有登記的寫入者，不表示已驗證或可發布」，
但狀態欄本身的文字仍容易被讀成「這件事完成了」。

**可能選項**
狀態欄改為複合式，明確分離三個維度：`寫入權` / `驗證狀態` / `發布狀態`。

---

### C4. Hero canonical 的定案狀態，文件與實際決策不同步

**現象**
`CLAUDE.md` 目前仍記載「正式 `index.html` 與 `mockups/home-attractor.html` 的
canonical Hero 來源未決，兩者都要保留，不得宣告任一為準」。
但依 2026-07-20 的紀錄，此事已定案為**以正式 `index.html` 為準**。

**影響**
代理讀 `CLAUDE.md` 會以為此事仍未決，可能重複提問或錯誤保留已作廢的稿件。

**可能選項**
更新 `CLAUDE.md` 該段，並在品牌總綱記錄裁決日期與結論。

1. 先處理 **A1**（登入驗證界線），因為它決定了所有技術回歸工作包的交付形狀。
2. 再處理 **A2**（完成的定義），因為它直接影響使用者能否驗收。
3. **A3** 與 **B2** 一起談，兩者都與「什麼情況下可以上正式站」有關。
4. **B1／B3／C1** 屬於規則寫在哪裡、寫得夠不夠明確，可一併補進 `AGENTS.md`。
5. **C2／C3／C4** 為工具與文件維護，成本低，可隨時處理。
