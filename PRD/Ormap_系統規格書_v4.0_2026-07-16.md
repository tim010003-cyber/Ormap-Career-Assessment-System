## 🌱 一、品牌定位(產品的靈魂,所有文案與功能設計的依據)

> 完整內容見品牌定位文件;此處為系統開發需要的濃縮版。**所有對外文案、Landing 介紹、模組說明,語氣與內容一律以此為準。**

- **核心精神**:「**先清醒,再勇敢**」。不鼓勵盲目追夢,而是陪伴一個人先理解自己、理解世界、評估現實限制與選擇風險,再勇敢做出屬於自己的人生與職涯決策。
- **服務定位**:一套深層生涯諮詢歷程,協助在職涯與人生中卡住、或正在尋找下一階段定位的人,重新理解天賦、性格、價值觀、核心信念、生存策略與現實處境。
- **服務邊界**:生涯諮詢,不處理創傷治療;心理狀態嚴重者轉介心理師/身心科。
- **服務路徑(小我 → 大我)**:
  1. **個人生涯設計**|理解自己,設計人生(Persona:生涯重整者/高功能工作者/市場錯位專業者)
  2. **專業與領導發展**|從做好自己,到影響他人(資深專業者、準管理職、中高階主管、HR、創業者)
  3. **組織與人才設計**|理解人才,設計合作(用人主管、HR、企業主:職務設計 → 人才標準 → 結構化面試)
  4. **組織與社會設計**|長期願景(關係共好)
- **護城河**:深度理解人 × 結構化分析 × 組織實務 × 人本價值立場——四者組合的轉譯能力。
- **倫理立場**:不服務操控人才的企業、不合理職缺包裝、只想找服從者的主管。
- **非主要對象**:在學學生/無工作經驗者、只要快速答案者、單點求職服務(純改履歷)、不願探索行動者、只追外部成功標準者。

## 🗂 二、資訊架構重整:三大分類

> 背景:網站設計老師回饋——現行 Dashboard 各功能**過於簡約、缺乏說明、太陽春**。v4.0 起所有模組進入三大分類,**每個模組都必須有清楚的說明架構**(這是什麼、幫你解決什麼、怎麼用、要花多久、產出什麼)。

| 分類 | 對應品牌服務路徑 | 現有模組 | 規劃中模組 |
|---|---|---|---|
| **1. 職涯** | 個人生涯設計(職涯面) | TWA 工作價值觀、HTI 何倫特質 | 履歷健檢、JD 評測 |
| **2. 生涯** | 個人生涯設計(人生面) | — | Super 人生彩虹(人生角色變換)等生涯評測 |
| **3. 組織** | 組織與人才設計 | — | 職務設計 → 結構化徵選標準整理工具(配合管理者課程) |

**外層(Landing/首頁)改版原則**:
- 進站先理解「原味藍圖是什麼、這是個什麼樣的工具網站、能幫你做什麼」,再進到各評測功能
- 首頁需要品牌定位介紹區(取材第一章),之後才是三大分類的工具入口
- 每個工具入口卡片:名稱 + 一句話說明 + 適合誰 + 預估時間(告別「陽春」)

## 🧱 三、兩階段商業規劃

### 第一階段:工具化與課程整合
- **授權碼(Code)機制**:單堂課程學員在課堂拿到授權碼,**一週內**可下載相關文件/範本(時效性門檻)
- **HR 範本與評測工具整合**:讓工具性質更多元(範本下載、課程配套工具)
- 優先方向:**把已成熟的諮詢內容轉成課程**,相關工具的說明與內容要做得更豐富完整(體驗優先)

### 第二階段:免費推廣與轉化
- 蒐集使用者對工具的意見與想法(回饋機制)
- 擴散給免費客戶 → 免費工具達到一定效果 → 慢慢導流到**付費課程**
- 部分功能維持免費、部分推付費版本

### 後台方向(從 CRM 到會員經營)
- 不再只是諮詢工具:要能看到**會員資料、需求與現況**(會員輪廓)
- 取得會員 Email(Google Auth 本已有)→ 後續 **EDM 行銷**
- 諮詢師工作流(白板/個案管理)保留,降為其中一個模組

## 🏗 五、技術架構演進(純靜態 → 需要後端)

**現況**(v3.8 第一章):純靜態 HTML + Firebase(Auth/Firestore/Hosting),無伺服器端程式。CI/CD 已於 2026-07-16 上線(push `main` 自動部署 Hosting;rules 仍手動)。

**v4.0 需求中,純靜態做不到或不安全的**:

| 需求 | 為什麼需要後端 | 建議積木 |
|---|---|---|
| 履歷健檢 / JD 評測 | AI API 金鑰不可放前端(會被盜刷);檔案解析需伺服器 | Cloud Functions + Claude API |
| 授權碼限時下載 | 前端驗證可被繞過;檔案需簽名網址 | Cloud Functions + Storage |
| EDM 發送 | 寄信本需後端(GAS 已廢,勿復刻) | Cloud Functions + 交易信服務,或第三方 EDM 平台 |
| 金流(未來付費版) | 收款回呼與訂單狀態必須伺服器端驗證 | Cloud Functions + 金流商(如 TapPay/綠界/Stripe) |

**原則**:留在 Firebase/GCP 生態系內加一層 **Cloud Functions**(同專案、按用量計費),不搬家、不重寫現有頁面。導入時機:第一個需要後端的功能開工時。
