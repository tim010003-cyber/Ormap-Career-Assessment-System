## 一、現有網站架構摘要（唯讀盤點）

### 1. 前端框架、路由及 UI 架構

- **已從程式碼確認**：純 HTML + 原生 JS ES Modules，無框架、無建置工具、無 `package.json`。每個 `.html` 是獨立頁面，`<script type="module">` 內直接 `import` Firebase SDK（gstatic CDN，鎖定 10.8.1）與站內共用模組。沒有前端路由器；頁面切換靠 `<a href>` 與 `location.href` 導頁。樣式為 Tailwind CDN + 每頁內嵌 `tailwind.config`；圖示用 Lucide 1.8.0（unpkg CDN）。品牌色：`primary` `#0d9488`、`secondary` `#0f172a`。
- **可重用**：`firebase-init.js`（唯一 Firebase 初始化入口，export `auth`/`db`）、`auth-utils.js`（`isCounselor`/`isSuperAdmin`/`migratePendingCounselor`/`pendingKeyForEmail`）、`config.js`（全站常數）。三者皆有清楚的檔頭註解慣例，新模組應延續此風格。

### 2. 後端框架與 API 架構

- **已從程式碼確認**：**完全沒有後端**。repo 內無 `functions/` 目錄、無任何 server 程式、無 `package.json`。所有「後端邏輯」目前都表現為 Firestore Security Rules（`firestore.rules`），由瀏覽器直接讀寫 Firestore。
- **推論**：這代表 AI App 規格要求的「AI 由後端呼叫、API Key 不進前端」「配額檢查在送出供應商請求前完成」「排程關閉」等，在現有架構上**沒有對應的技術積木**，必須新增 Cloud Functions（或等效的伺服器端執行環境）才能實作，不是「調整既有後端」而是「從零建立後端層」。
- **佐證**：`PRD/Ormap_系統規格書_v4.0_2026-07-16.md` 第五章已明確寫出「純靜態做不到或不安全的需求」清單，其中「履歷健檢／JD 評測」「授權碼限時下載」都標示「⚠️ 需後端 + AI API」「⚠️ 需後端驗證」，建議積木為「Cloud Functions + Claude API」。本次職務設計 App 面臨完全相同的技術缺口，且 `PRD/需求池.md` 第四節已把「職務設計工具」列為需求池項目之一（前置：課程內容定調），方向與本次任務一致。

### 3. 資料庫、ORM、Migration 方式

- **已從程式碼確認**：資料庫為 Cloud Firestore（NoSQL 文件資料庫）。無 ORM、無 Migration 工具，Schema 只隱含在程式碼的讀寫呼叫與 `firestore.rules` 之中，沒有集中的 Schema 定義檔。
- **推論**：新功能的資料模型需要自行設計、自行維護一致性（沒有 Migration 框架可用），且目前沒有任何機制驗證「寫入 Firestore 的資料形狀」是否符合預期（前端直接 `setDoc` 什麼形狀就寫什麼形狀）。這對本 App 規格中大量的「固定欄位 ID、型別、Schema 驗證」要求是一個要特別設計的落差。

### 4. 既有會員、登入、Session 與權限機制

- **已從程式碼確認**：登入只有 Firebase Auth Google 一種方式（`GoogleAuthProvider` + `signInWithPopup`）。Session 由 Firebase Auth SDK 內建管理（瀏覽器本機持久化），沒有自訂 Session／Cookie 機制。權限判定完全依賴 `counselors/{uid}` 文件的 `isActive`／`isSuperAdmin`／`isDeleted` 欄位，`auth-utils.js` 提供 fail-safe（任何錯誤回 `false`）的判定函式。
- **推論**：目前只有「一般已登入使用者」與「諮詢師／Super Admin」兩層角色，沒有「梯次（cohort）」「限時邀請」「開放/截止時間」等概念，這些都是本 App 需要新增的資料與邏輯。

### 5. 是否已有講師／管理者與一般使用者角色

- **已從程式碼確認**：有——`counselors/{uid}` 驅動的諮詢師／Super Admin vs. 一般使用者兩層角色模型，是目前站內最接近「講師／學員」的既有樣式，值得沿用其**設計模式**（fail-safe 判定函式、以 Firestore 文件驅動角色、pending-by-email 邀請後遷移到 uid）。
- 但**不建議直接重用 `counselors` 這個集合本身**：它的語意是「生涯諮詢師」，與「職務設計課程講師」是不同角色體系，混用會讓兩個產品邊界糾纏在一起（例如一個生涯諮詢師不應該自動變成職務設計課程的講師，反之亦然）。建議另立語意獨立的新集合（見第五節）。

### 6. Email 邀請、魔法連結或測試帳號能力

- **已從程式碼確認**：**沒有任何 Email 寄送能力**。原本的 Google Apps Script（GAS）轉信功能已於 2026-07-16 整體移除（`CLAUDE.md` 與 `PRD/需求池.md` 皆有記載），現行邀請流程改為「複製邀請連結」人工轉發，沒有魔法連結（magic link）驗證，也沒有測試帳號機制——所有登入都必須是真實 Google 帳號。
- **推論**：本 App 規格中的「邀請學員」也必須採複製連結模式（例如：講師建立梯次 → 產生一組邀請碼/連結 → 對方用 Google 帳號登入 → 系統比對 email 是否在受邀名單並完成綁定），不能假設有 Email 寄送管道。這與規格套件已確認的「不重新引入 GAS」原則一致，且待確認決策 D-04「Beta 是否需要 Email 提醒」在現有架構下答案傾向「否，先站內提醒」。

### 7. 檔案儲存與下載方式

- **已從程式碼確認**：`firebase-init.js` 的 `firebaseConfig` 雖然帶有 `storageBucket` 欄位（Firebase 專案預設會有這個值），但**全站沒有任何一處 `import` Firebase Storage SDK 或呼叫 `getStorage`**——Storage 目前完全未被使用。目前唯一的「下載」行為是瀏覽器端 CSV/文字產生（未見任何檔案上傳/下載邏輯）。
- **推論**：Word 匯出若採「Cloud Function 產生 → 上傳到 Storage → 回傳簽名網址下載」的做法，會是這個專案第一次真正使用 Storage，需要新增對應的 Storage Security Rules（目前 repo 內也沒有 `storage.rules`）。

### 8. 是否已有 Word 或其他文件產生能力

- **已從程式碼確認**：**完全沒有**。已針對 `docx`／`mammoth`／`officegen` 等關鍵字搜尋全站程式碼與文件，無任何匹配；也沒有 `jsPDF`／`pdfmake` 之類的 PDF 產生邏輯。`PRD/Ormap_系統規格書_v4.0_2026-07-16.md` 亦註記「PRD v3.7 docx 已由 v3.8/v4.0 接替（docx 封存不再維護）」，代表過去也沒有動態文件產生的先例，只有靜態存放的規格文件本身是 `.docx`。
- 這代表 Word 產生（`.docx` 樣板套用、章節/表格/分頁邏輯、繁中字型）是**全新能力**，沒有既有程式可以參考或延伸。

### 10. API Key 與秘密管理方式

## 三、建議整合方式

### 3. 建議 App 放置方式

**建議：同一個 GCP 專案（`careervalue-ormap`）、同一個 Firebase Auth 身分系統、同一個 Firestore 預設資料庫（新增獨立前綴的集合），但用 Firebase Hosting 的「多站台（multi-site hosting）」建立第二個 Hosting target，搭配獨立的 Cloud Functions 與獨立的 CI workflow。**

理由：
- 規格套件已確認「App 在官網提供入口，但 Beta 優先採獨立模組與獨立部署」「不強制共用官網全部會員與資料庫，但應避免重複且不安全的身分系統」。
- 完全獨立的 GCP 專案雖然隔離度最高，但會被迫重建一套身分系統（違反「避免重複且不安全的身分系統」），且需要額外的帳單/IAM 設定，對 Beta 規模是過度投資。
- 同一 Hosting site、同一個 `firebase deploy --only hosting` 的做法，會讓 App 每次部署都跟正式站綁在一起（任何一邊出錯都互相影響），不符合「獨立部署」的產品決策。
- 「多站台 Hosting + 共用 Auth/Firestore + 獨立 Functions/CI」是折衷點：身分不重複、資料用前綴集合隔離、部署互相獨立，且全部落在 Firebase 既有生態系內（呼應 `Ormap_系統規格書_v4.0` 第五節的既定技術方向：「留在 Firebase/GCP 生態系內加一層 Cloud Functions，不搬家、不重寫現有頁面」）。

此為建議方案，非唯一解——是否要進一步隔離到子網域或獨立 GCP 專案，屬於待確認決策（對應規格套件既有的 D-12），列入第九節。

### 4. 建議 URL 或路由

- **App 路徑**：`tools/job-design/` 底下（沿用既有 `tools/twa.html`、`tools/hti.html` 的目錄慣例，內部 import 改用 `../../` 相對路徑）。頁面仍是「每頁一個獨立 HTML」的既有模式，例如 `tools/job-design/cases.html`（案例列表）、`tools/job-design/case.html?caseId=...`（單一案例的模組填寫/確認/輸出介面，用 query string 切換目前模組，避免又建一堆檔案）、`tools/job-design/admin.html`（講師梯次管理）。
- **對外網址**：若採第 3 點的多站台方案，可選：(a) 同網域路徑 `https://careervalue-ormap.web.app/tools/job-design/`（最簡單，不需額外網域設定）；或 (b) 第二個 Hosting site 產生的獨立 `.web.app` 網址（例如 `ormap-jobdesign.web.app`）。**建議先採 (a) 同網域路徑**：對使用者體驗更一致、不需要額外 DNS/網域設定，且「獨立部署」的訴求已經由「獨立 Hosting target + 獨立 Functions + 獨立 CI」滿足，不一定需要獨立網域。子網域與否留待 PO 確認（同樣對應 D-12）。

```
  │                                    explicit/organized/inference/gap 內容
  ├ document_versions/{docType}     — job_description／selection_design／public_jd
  │   └ versions/{versionNo}          三份正式輸出各自的版本歷史
  ├ confirmations/{confirmId}       — 人工確認紀錄（欄位/區塊/文件層級皆可指向）
  ├ instructor_feedback/{feedbackId}— 講師回饋與退回紀錄
  └ source_links/{linkId}           — 上下游欄位追溯（供 ImpactFlag 計算查詢）
jd_word_exports/{exportId}          — 合併 Word 產出紀錄：case_id、來源版本、
                                        產生時間、下載確認時間（頂層集合，方便管理頁彙整查詢）
jd_usage_ledger/{logId}             — 每次 AI 呼叫的用量明細（頂層，供配額/成本查詢）
jd_quota_counters/{cohortId_uid_date} — 用量計數器（Cloud Function 交易式遞增，
                                        AI 呼叫前先讀這裡而非掃整個 ledger）
jd_audit_events/{eventId}           — 關閉／重開／權限相關的稽核事件
```

**設計取捨說明**：
- `FieldDefinition`（欄位字典）**不建議存成 Firestore 集合**，而是以程式碼形式（JSON/JS 常數檔，直接對應 `02_欄位字典_v1.md` 的 `field_id`）同時交給前端表單驗證與 Cloud Functions 的 Schema 驗證使用——欄位字典改版頻率低、需要嚴謹的 code review 與版本紀錄，用 git 版控比用可被誤寫的 Firestore 文件更安全，也避免「AI 或前端不小心寫壞欄位定義」的風險。
- `AccessWindow` 不獨立建集合，直接把 `opens_at`／`submission_due_at`／`first_final_download_at`／`scheduled_close_at`／`closed_at`／`reopened_until`／`reopen_reason` 當作 `jd_cases` 文件的欄位（1:1 對應案例），梯次層級的預設值放在 `jd_cohorts`。這樣關閉排程只需要查詢 `jd_cases`，不必再 join 另一個集合。
- `SourceLink` 保留為獨立子集合（而非完全內嵌在欄位值裡），因為「上游修改時要找出所有受影響的下游欄位」是一個需要查詢的操作（by 來源欄位 ID），內嵌在每個欄位值裡會讓這個查詢變成要掃描整份案例的 JSON，獨立集合可以用 `where('source_field_id', '==', ...)` 直接查。

### 6. 欄位 ID 對應策略

`02_欄位字典_v1.md` 的每個 `field_id`（如 `m1.trigger_event`、`m3.competency_anchors`）**直接作為 `module_submissions/{moduleCode}.structured_values` 這個 JSON 物件的 key**，不另外為每個欄位建立獨立 Firestore 文件（field_id 有近 150 個，逐欄位建文件會讓「儲存一個模組」變成上百次寫入，且失去模組層級的原子性）。

每個欄位值統一用以下物件形狀儲存（滿足欄位字典要求的「原始輸入／AI 整理／AI 推論／人工確認分開保存」）：

```json
{
  "value": "...",
  "source_type": "user_input | ai_organized | ai_inference",
  "ai_job_id": "ai_job_xxx",
  "confirmed": false,
  "confirmed_by": null,
  "confirmed_at": null,
  "version": 1
}
```

「重複清單／重複物件」型別的欄位（如 `m3.work_items`、`m5.interview_questions`）存成**陣列，陣列中每一項各自帶有上述 source/confirmed 中繼資料**，滿足「重複清單與表格不可用單一長文字取代，否則無法追溯與局部重做」的實作檢查要求。AI 回應中的 `source_map` 由 Cloud Function 解析後，寫入對應欄位的 `ai_job_id` 與（若跨欄位）獨立的 `source_links` 子集合紀錄，供第五節的 ImpactFlag 查詢使用。

### 7. 登入與權限方案

- 沿用 Firebase Auth Google 登入（同一身分，不重建帳號系統）。
- 新增 `jd_users/{uid}` 文件，欄位：`role`（`instructor`/`student`）、`cohort_ids[]`（可能屬於多個梯次）、`isActive`、`invitedByEmail`（pending 用）。仿照 `auth-utils.js` 模式提供 `isJdInstructor(db, uid)`、`isJdCohortMember(db, uid, cohortId)`，一律 fail-safe 回傳 `false`。
- 邀請流程沿用「複製連結」+ `pendingKeyForEmail` 的 email-pending-migrate 手法（第一節第 6 點已確認站內沒有寄信能力，必須維持此模式，不重新引入 GAS）。
- **與既有站台的關鍵差異**：既有 Rules 中有多處「只要登入即可 `read`/`list`」的寬鬆模式（例如 `whiteboard_sessions`、`counselors` 的 read），這類寫法對本 App 不適用——規格明確要求「學員只能查看自己的案例」「講師只能查看其有權管理的梯次與案例」，因此 `jd_cases` 等集合的 Rules 必須逐筆比對 `owner_uid` 或講師的 `cohort_ids` 交集，不能沿用「登入即可讀」的寫法（現有 `assessments` 集合的「A2 殘留」問題——已登入者仍可整表 `list`——是一個要避免重蹈覆轍的前例）。
- AI 呼叫與 Word 產生一律走 Cloud Functions callable function（用 Firebase Auth ID Token 驗證），不透過前端直接寫入 Firestore 產生「AI 結果」——確保配額/期限檢查一定會被執行到，不能被繞過。

### 8. AI Provider 抽象方案

- Cloud Functions 內建立 `AiProvider` 介面（`complete(taskCode, input, schemaVersion) → 結構化輸出`），至少兩個實作：`MockAiProvider`（規則式/樣板式回應，不呼叫任何外部 API，用於 Phase 1–2）與之後的 `AnthropicProvider`（Phase 3 才接上真實 API Key）。
- 任務登記表（task registry）以程式碼維護，每個 `task_code`（如 `M1-01`、`M3-08`）對應：提示詞版本檔案、輸出 JSON Schema（建議用 Zod 或 AJV 驗證）、允許讀取的已確認上游欄位清單、是否為高成本任務（需上游確認後才可執行）。
- 不把整份六份 Markdown 送給模型：每次呼叫只組裝「本任務規則 + 已確認上游摘要（依欄位字典的 source_type=confirmed 篩選）+ 本次原始輸入」，對應 `04_AI任務規格_v1.md` 的共通輸入契約。
- Schema 驗證失敗 → 最多自動修復一次 → 仍失敗則轉人工並保留原始輸入（不寫入 `structured_values`，只記錄 `ai_jobs` 的失敗狀態與原文）。
- API Key 用 Google Secret Manager／Firebase Functions v2 secret 綁定，只綁定給需要呼叫供應商的那個 function，正式 Key 要到 Phase 3、PO 明確同意後才設定。

### 9. Word 產生方案

- Cloud Function（`generateWordExport`）伺服器端用 Node.js 的 `docx` 套件（程式化組裝章節/表格/分頁，不使用 AI 產生排版），依 `05_Word輸出規格_v1.md` 的固定章節順序（封面 → 文件資訊 → 思考歷程 → 三份正式輸出 → 待確認事項 → 版本摘要）讀取已凍結的 `document_versions` 與已確認欄位值組裝內容。
- 產生的 `.docx` 上傳到 Cloud Storage 私有路徑（新引入 Storage 用途，需新增 `storage.rules`），回傳短效期簽名網址供下載，滿足「授權碼限時下載」等未來功能也可能共用同一套「簽名網址下載」機制的延展性。
- **兩段式下載確認**（呼應「產生失敗或下載中斷不得開始三天倒數」）：
  1. `generateWordExport` 產生檔案並記錄 `WordExport` 草稿狀態，回傳下載網址；
  2. 前端瀏覽器成功觸發下載後，呼叫另一個小 function `confirmWordDownload`，此時才寫入 `access.first_final_download_at` 並計算 `scheduled_close_at = now + 72h`。
  只有第 2 步真正執行，倒數才會開始，避免產生成功但下載失敗時誤啟動關閉倒數。

### 10. 72 小時關閉與排程方案

- 用 Firebase Functions v2 的 `onSchedule`（背後即 Cloud Scheduler）建立排程 function，建議每 15–30 分鐘跑一次：查詢 `jd_cases` 中 `scheduled_close_at <= now` 且 `closed_at == null` 的文件，逐筆用 Firestore transaction 設定 `closed_at`／`status = 'closed'` 並寫入 `jd_audit_events`——transaction 內先檢查 `closed_at` 仍為 `null` 再寫入，確保排程重複執行（例如重疊觸發）不會有副作用，符合「排程重複執行必須具備冪等性」的要求。
- 所有時間戳記存 Firestore Timestamp（UTC），前端一律以 Asia/Taipei 顯示；關閉判斷只信任伺服器時間（Cloud Function 與 Firestore Rules 端），不信任使用者本機時間。
- 講師重開：`reopenCase(caseId, reason, until)` callable function，寫入 `reopened_at`／`reopened_until`／`reopen_reason` 與稽核事件；狀態機依 `03_資料與狀態模型_v1.md` 第七節的 `closed → reopened` 轉換。
- **雙層強制**：Firestore Rules（擋掉客戶端對已關閉案例的直接寫入）與 Cloud Functions（AI 呼叫/Word 產生前，各自再次獨立檢查案例狀態與配額）都要各自檢查一次，不能只靠前端判斷是否顯示按鈕。
- **前提**：Cloud Scheduler／Cloud Functions 需要 Firebase **Blaze（用量計費）方案**，第一節第 11 點已指出目前方案別未能從程式碼確認，且備份功能也還「待 PO 在 Console 設定」，暗示可能尚未升級——這是本方案最優先要確認的基礎建設前提（見第九節）。

### 11. 安全及隱私風險

- **API Key 外洩**：靠 Secret Manager + 只在 Cloud Functions 內使用來源阻絕；Phase 1–2 用 Mock Provider 完全不需要真實 Key，降低早期風險。
- **跨案例/跨梯次資料外洩**：新集合的 Rules 必須逐筆比對擁有者/梯次成員，不可沿用現有 `assessments` 集合「已登入即可 list」的寬鬆前例（該問題目前仍是本站已知未修復的技術債，列在 `PRD/需求池.md` 的 A2 殘留）。
- **敏感內容外洩到 JD**：規格已定義 QA-02 揭露檢查任務，Word 產生前與 JD 確認前都必須跑過這道檢查並阻擋不合格的發布。
- **資料保留與刪除**：待確認決策 D-05／D-06 尚未拍板，正式上線前必須有明確政策，本方案暫先只做「關閉」（禁止編輯/AI），不做刪除。
- **Log 衛生**：Cloud Functions 的錯誤處理不可把使用者原始口述內容、AI Key 或完整 Prompt 印進日誌；建議統一用結構化 log 並對長文字欄位做截斷。
- **未授權呼叫/濫用成本**：所有 callable functions 一律檢查 `context.auth` 是否存在且屬於有效梯次成員，避免任何人繞過前端直接打 Cloud Functions 端點消耗 AI 配額；必要時可加 Firebase App Check。
- **Firestore 免費額度/新方案的帳單風險**：一旦導入 Cloud Functions + Scheduler，必須升級 Blaze 方案（用量計費），需要 PO 在啟用前確認預算與配額上限（第八節已規劃配額表，但實際數字待 D-08 決策）。

| Phase 0 基礎建設 | 3–5 天（含 Blaze 升級、Functions 骨架、多站台 Hosting、講師/學員角色與邀請流程） |
| Phase 1 系統骨架 | 4–6 天（案例模型、路徑 B 基礎確認頁、狀態機、權限鎖定） |
| Phase 2 路徑 B + Mock AI + Word + 72h 關閉 | 10–15 天（M3–M6 頁面與確認 UI、Mock Provider、Word 產生服務、排程 function、來源追溯 UI） |
| Phase 3 正式 AI API | 5–8 天（供應商串接、Secret 管理、提示詞版本化、配額/重試、品質測試） |
| Phase 4 路徑 A | 6–9 天（M1、M2 頁面、完整上下游追溯與 `needs_review` 重做流程） |
| Phase 5 驗收 | 3–5 天（真實案例測試、清單逐項驗收、修正） |

### 主要風險

1. **從零建後端是本案最大工程量來源**：現有站台完全沒有 Cloud Functions 前例，Phase 0 的架構決策（多站台 Hosting、Secret 管理、排程冪等性）品質會直接影響後續所有階段，值得在 Phase 0 花較多時間做對，而不是壓縮。
2. **Blaze 方案與帳單核准是外部依賴**：若 PO 尚未準備好升級付費方案，整個技術路線會卡住，建議此問題最優先確認（見第五節第 1 點）。
3. **Word 產生的繁中排版品質**：全新能力、無既有程式可參考，`05_Word輸出規格_v1.md` 對表格跨頁、字型、標題階層有具體要求，建議 Phase 2 及早找 1–2 個真實內容案例測試，避免累積到 Phase 5 才發現排版問題。
4. **與其他已規劃功能（履歷健檢／JD 評測）共用 Cloud Functions 基礎設施的機會與風險**：`Ormap_系統規格書_v4.0` 已把這兩個功能與本案列在同一份「需要後端」清單，Phase 0 建立的 Functions 專案骨架、Secret 管理、Provider 抽象層可以刻意設計成三者共用，降低重複建設成本，但也要注意本案的 Beta 期限/關閉機制屬於本案專屬邏輯，不要為了共用而把不相關的產品耦合在一起。
5. **Schema 驗證與 AI 忠實度是品質風險而非純技術風險**：規格對「不補造」「上下游一致」「揭露邊界」的要求很細，Mock Provider 階段測不出這些問題，Phase 3 導入真實 API 後才會真正暴露，建議 Phase 3 提前規劃 `04_AI任務規格_v1.md` 第七節的最小測試集，而不是等 Phase 5 驗收才做。

## 七、建議的第一個垂直切片（具體化）

呼應使用者訊息「第一個垂直開發切片優先採路徑 B」的決定，具體建議切片範圍為：

**一個講師、一個學員、一個既有職務優化案例，從登入到 Word 下載的完整端到端流程**：

1. 講師登入 → 建立測試梯次（開放/截止時間、AI 配額先給寬鬆預設值）→ 產生邀請連結。
2. 學員用連結登入 → 建立案例（路徑 B）→ 完成「既有職務基礎確認」（純人工欄位，`existing.*`）。
3. 系統跑 `PATH-01`（Mock）完整度檢查 → 進入 M3 職務與人才規格：逐章節口述輸入 → Mock AI 整理（`M3-01`～`M3-09`）→ 人工逐項確認（含五級行為錨點、五維成熟度最低/理想等級）。
4. 產生 M4 職務說明書草稿（Mock `M4-01`）→ 人工確認。
5. 產生 M5 甄選流程設計書草稿（Mock `M5-01`～`M5-03`）→ 人工確認。
6. 產生 M6 JD 對外文案（Mock `M6-01`）→ 跑 `QA-02` 揭露檢查（Mock）→ 人工確認發布前檢查。
7. 產生合併 Word（含思考歷程 + 三份正式輸出 + 待確認事項 + 版本摘要）→ 成功下載 → 記錄 `first_final_download_at` → 排程 72 小時後自動關閉 → 講師可重開。

此切片刻意**全程使用 Mock AI**，目的是先驗證「資料模型、確認節點、來源追溯、Word 產生、期限關閉」這些之後不會因為換成真實 AI 供應商而改變的核心機制，把最貴、最需要 PO 額度核准的「真實 AI 呼叫」延到 Phase 3 才做，符合使用者「第一階段不要實際呼叫付費 API」的明確指示。
