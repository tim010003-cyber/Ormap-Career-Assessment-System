## 3. 剛修完、尚未驗證的兩件事

### 3.1 靜態檔快取（`f8c2e56`）

Firebase 預設 `max-age=3600`，而這個專案**檔名不帶版本雜湊**，所以部署後使用者會拿到舊檔一小時。症狀是「新功能 is not a function」，看起來像程式壞掉。

已改為 `html/js/css/json → no-cache`（圖片字型維持長快取）。

**待驗證**：部署完成後確認標頭
```bash
curl -sI "https://careervalue-ormap.web.app/tools/job-design/jd-ai.js" | grep -i cache-control
```
應為 `no-cache`。若仍是 `max-age=3600`，代表 headers 設定沒生效。

### 3.2 授權碼頁的初始化保護（`4e763e8`）

原本 `onAuth` 失敗被 `.catch(() => {})` 吞掉，畫面永遠停在「載入中…」。已改為顯示原因＋重新載入按鈕＋12 秒逾時。

**待驗證**：實際走一次「後台 → 授權碼 → 產碼 → 兌換 → 用一次 AI → 次數 -1」。

## 6. 這個工作階段踩過的坑（別重蹈）

1. **分支被切走而不自知**
   另一個工作階段把 HEAD 切到 `fix/twa-content-and-zero-state`，我後續 5 個提交都落在那條分支，卻以為還在原分支。第一次合併只帶進一半，是靠部署後檢查檔案才發現。
   → **合併前先 `git branch --show-current` 與 `git log --oneline origin/main..HEAD` 對照。**

2. **`firebase.json` 是純 JSON，不能寫註解**
   我加了 `/* */` 說明導致解析失敗；接著用正則移除註解時，`node_modules/**` 裡的 `/**` 被當成註解開頭，把整個 ignore 清單吃掉。
   → **改 `firebase.json` 用程式讀寫物件，不要手改也不要用正則。**

3. **測試看起來過了不代表驗到東西**
   併發配額測試一度顯示「全部通過、用量 0」，看似漏洞，實際是退費機制生效讓測試失去意義。改成直接測 `consumeQuota` 才驗到上限。

4. **訊息要出現在操作點旁邊**
   AI 不可用的說明原本放在頁面頂端，使用者按下方按鈕只感覺到「畫面往上跳一下」。改成貼著按鈕（12px）才有效。

5. **登記簿沒同步會誤導後續判斷**
   我曾依「已確認／未修復」把四個 P1 當成前置條件，實際上 `SEC-FIX-D` 早就修好也部署了。→ 修完務必回頭更新登記簿。

## 7. 核心原則（不要違反）

**AI 是整理者，不是審核者。只提醒，不阻擋。**
唯一的硬限制是配額用完。不要加回下載閘門、文件狀態機、未確認即禁止發布這類設計——使用者明確否決過，而且我踩過三次。

**沒有 AI 就不整理，不用 Mock 頂替。**
Mock 是關鍵字規則產生的，品質差到會誤導人。被擋的只有「口述整理」；權責矛盾偵測、文件組裝、揭露與歧視提醒都是規則邏輯，照常運作。
