# 原味藍圖｜跨代理協作契約

> 版本：v1.2
> 日期：2026-07-20  
> 狀態：**提案／部分暫行條款已由使用者於 2026-07-21 確認**
> 適用對象：使用者、Codex、Claude 與後續加入本專案的其他代理  
> 文件目的：建立中立、可追溯、可驗收的合作方式；不以單一代理的偏好決定品牌、產品、技術或發布結果。

---

## 1. 契約目的

本契約用來回答五件事：

1. 一項工作由誰提出、誰實作、誰驗收、誰決定發布。
2. 產品／體驗與工程／營運意見衝突時，如何處理。
3. 多個代理如何避免同檔衝突與回歸。
4. 如何把介面、資料、內容、狀態與驗證證據交接清楚。
5. 如何讓使用者保有品牌與產品決策權，又不需要親自處理每個工程細節。

本契約不取代產品 PRD、品牌總綱或技術規格；它只規範這些文件如何被多人共同執行。

## 2. 已確認、提案與尚未驗證

### 2.1 已確認

- 使用者擁有品牌定位、產品範圍、工具正式名稱、成熟度、價格、主要 CTA、頁面增刪與對外發布的最後決策權。
- 專案目前需要同時保護既有評測功能，並逐步完成 Index、Dashboard、職務設計、內容與後續課程架構。
- 不同代理能力側重不同：產品／體驗設計與既有評測／Cloud 工程都需要明確責任與互相驗證。
- 工作區可能同時存在使用者或其他代理的未提交變更，任何代理都不得擅自覆蓋或還原。
- Repository 已有 PR preview workflow；`main` 收到新提交時，merge workflow 會直接部署 Firebase Hosting live channel。工作流程是否能在 GitHub 實際成功執行及完整回復仍需部署任務驗證。
- 已確認的存取控制、權限語意與索引版本控制問題統一登記於 `PRD/原味藍圖_已知缺陷與風險登記簿.md`；登記不代表已修復。

### 2.2 本文件提案

- 以「角色」分配責任，再把角色指派給當次最合適的代理；不設永久的全專案代理統治權。
- 產品驗收與技術驗收分開，重大變更兩者都通過後才可送交使用者決定發布。
- 發布執行權不等於產品決策權，也不等於可單方面否決另一專業領域。
- 每個工作包設一位寫入負責者；審查者原則上不直接在同一檔案同步修改。

### 2.3 尚未驗證／待決策

- 正式 `index.html` 與 `mockups/home-attractor.html` 何者是 Hero canonical 文字來源，仍待使用者明確裁決。
- PR preview 與 main live workflow 的設定檔已存在，但 GitHub 權限、preview 實際產生、失敗處理及回復流程尚未驗證；Firestore Rules 仍不在 Hosting workflow 的自動部署範圍。
- 評測／白板首批 DOM hook、狀態、路由與資料介面已建立初步清單，但尚待 Claude 技術確認；共用 design tokens 與完整評測外殼契約仍未完成。
- 本契約需由使用者確認後，才升為「已定稿」並據此調整舊文件中的身分式責任描述。

### 2.4 2026-07-21 已確認的暫行協作決策

- 使用者允許代理在工作包指定的預覽環境執行登入後前臺測試。工作包仍須列出確切網址、測試身分取得方式與資料操作邊界；不得把密碼、token、cookie 或其他憑證寫入對話、文件、日誌或版控。
- 前臺／產品體驗測試可由 Codex 執行，工程、資料、權限與發布前技術檢查可由 Claude 執行；這是能力導向預設，不是永久排他權，工作包可另行指定。
- 使用者驗收以可實際開啟的預覽網址為最低交付方式；只有本機或工作區成果時，必須明確標示使用者目前無法查看。
- 交付採多維狀態，不再以單獨的「已完成」概括實作、Git、驗證與發布。整體工作包是否最終完成，由使用者確認。
- 實作者可將範圍內成果提交到非 `main` 分支保存與審查；需要推送、PR 或預覽時，工作包須指定預覽執行者。合併 `main` 仍只在使用者核准正式發布後執行。
- Firebase Emulator 與種子資料可作為降低登入後回歸風險的方向；是否導入、資料模型與成本須另立工程工作包評估，本次決策不等同已授權安裝、修改 Firebase 設定或建立測試資料。

## 3. 角色與權責

### 3.1 使用者／產品決策者

負責：

- 品牌定位、核心承諾、主要與次要受眾。
- 工具名稱、用途、成熟度、免費／付費邊界與價格。
- 服務、課程、諮詢及組織方案的範圍。
- 主要 CTA、重大頁面增刪及資訊架構方向。
- 可接受的風險與正式對外發布。

不需要負責：

- 每個程式細節、測試寫法或部署指令。
- 在產品與工程意見一致且不涉及新決策時，重複確認已定稿內容。

### 3.2 產品與體驗負責者（Product & Experience Owner）

負責：

- 平台與產品資訊架構、網站地圖與使用者旅程。
- Index／Dashboard 分流、公開網站、導覽與內容層級。
- 內容模型、區塊 ID、命名、CTA、狀態呈現與產品一致性。
- UI／UX、設計系統、響應式、可及性與產品驗收。
- 把使用者已確認內容轉成可實作規格；不得自行改變品牌或產品決策。

### 3.3 工程與營運負責者（Engineering & Operations Owner）

負責：

- 技術可行性、資料模型、Auth、Firestore、權限與安全。
- CI／CD、預覽環境、部署、監控、回復與技術驗收。
- 評估變更對資料、效能、相容性及既有功能的影響。
- 確認秘密值不進入前端、文件、差異或日誌。
- 對無法安全發布的技術風險提出阻擋證據與替代方案。

### 3.4 評測／功能模組負責者（Module Owner）

負責：

- 指定模組的實作、資料與行為正確性。
- 保護題目、計分、結果映射、草稿、儲存、報告與跨頁流程。
- 依工作包提供自我驗證、回歸證據及已知限制。
- 不擴張到未授權的全站架構、品牌內容或其他模組。

### 3.5 發布執行者（Release Operator）

負責：

- 核對發布版本、核准紀錄、預覽驗收及回復點。
- 執行已獲核准的合併與部署。
- 回報實際部署結果、版本、時間與未完成項目。

發布執行者不得因持有部署權限而自行改變產品內容、縮減驗收範圍或替使用者作發布決策。

- Codex 不因負責產品架構而自動取得 Cloud、資料或發布的單方決定權。

- 審查意見必須指出依據、風險、影響與可驗證的通過條件，不能只用「風格偏好」或「我主責」作為否決理由。
- 超出角色權責、會改變已確認產品方向或雙方無法依證據收斂時，由使用者裁決。

## 5. 驗收與否決邊界

### 5.1 產品驗收

檢查：

- 是否符合已確認品牌與產品文件。
- 使用者旅程、命名、CTA、狀態及下一步是否一致。
- 桌機、平板、手機的內容層級與操作是否合理。
- 鍵盤操作、焦點、語意標題、對比、替代文字與錯誤訊息。
- Demo、開發中與規劃中功能是否被清楚區分。

產品驗收可以阻擋：會誤導使用者、違反已確認產品決策、造成重大流程斷點或可及性失效的變更。

### 5.2 技術驗收

檢查：

- 計分、資料、Auth、Firestore、權限與既有流程是否回歸。
- 錯誤、載入、儲存、恢復、重試及權限拒絕是否可處理。
- CI、預覽、部署、Rules／索引及回復是否一致。
- 是否洩漏秘密、擴大資料存取或引入未核准的外部成本。
- 正式 Hosting 是否混入 mockup、Demo、測試資料或內部頁。

技術驗收可以阻擋：資料損壞、權限錯誤、安全風險、核心功能回歸、無法回復或未經核准的正式部署。

### 5.3 不構成合理阻擋的情況

- 未連結規格或證據的個人偏好。
- 要求另一角色重做已通過且未受本次變更影響的工作。
- 以持有程式碼、部署權限或文件作者身分主張永久所有權。
- 把尚未驗證的推論描述成已知缺陷。

### 5.4 意見衝突處理

1. 各方先寫出問題、證據、影響、嚴重度與建議條件。
2. 判斷屬於產品決策、技術風險、跨域問題或單純偏好。
3. 能以測試或預覽驗證者，先用最小實驗取得證據。
4. 涉及品牌、範圍、價格、主要 CTA、頁面增刪或風險接受度時，交由使用者決定。
5. 裁決後更新決策紀錄；未更新前不得以口頭推論擴大影響範圍。

| 保護行為 | 不得退化的既有功能 |

| 回復方式 | 失敗時如何恢復且不破壞他人變更 |
| 狀態 | 分別記錄寫入權、實作、Git、驗證、可見環境／發布，不以單一欄位混合 |

### 6.2 單一寫入者規則

- 寫入前先在 `PRD/原味藍圖_跨代理交接與檔案占用表.md` 登記具體路徑、工作包與唯一寫入者。無法確認占用狀態時，只可唯讀。
- 同一檔案同一時間只有一個寫入負責者。
- 審查者先以 comment、差異或交接紀錄提出修改，不直接同步搶寫。
- 需要換手時，先記錄目前狀態、未完成項目、工作區差異與驗證結果。
- 發現非本工作包變更時，保留原狀；不得 reset、checkout、覆寫或順手整理。
- 同一功能分散於多個 HTML、CSS、JS 或資料檔時，工作包必須列出完整影響面，不能只以主要頁面作範圍。

## 7. 介面契約最低要求

評測、報告、時間線與白板的首批現況介面記錄於 `PRD/原味藍圖_評測與白板介面保護清單_v1_2026-07-20.md`。該清單在 Claude 完成技術確認前屬初步盤點，不足以授權跨結構重寫。

跨代理或跨模組交付至少要定義：

- 路由與頁面用途。
- 區塊穩定 ID 與繁體中文名稱。
- DOM 掛載點、不可更名的 class／data attribute／事件名稱。
- 模組輸入、輸出、狀態與錯誤格式。
- Auth 狀態、角色、權限與未登入返回行為。
- Firestore collection／document 的用途與讀寫責任；不得在文件記錄秘密值或真實個資。
- CTA 文字、目的地、顯示條件及不可用狀態。
- 桌機、平板、手機與鍵盤操作差異。
- 需要執行的自動或人工回歸項目。

若上述介面尚未整理，先做唯讀盤點與最小契約，不直接進行跨全站重構。

## 8. 高風險區與保護方式

分類只用來決定保護強度；具體已知問題必須逐項登記於 `PRD/原味藍圖_已知缺陷與風險登記簿.md`，包含 ID、證據、嚴重度、狀態、負責角色與驗收條件。未登記的已知缺陷視為追蹤流程缺口，不得只停留在對話或程式註解。

| 區域 | 主要風險 | 最低保護要求 |
|---|---|---|
| TWA／HTI | 題目、計分、結果映射退化 | 固定測試案例、資料版本核對、完成流程回歸 |
| 草稿與儲存 | 遺失、覆蓋、無法恢復 | 未登入／已登入、重整、錯誤與恢復測試 |
| 報告／時間線／白板 | 跨頁資料或 mirror mode 失效 | 保護 DOM hook／旗標；逐頁回歸 |
| Auth／Firestore／Rules | 越權、靜默失敗、前後端不同步 | 權限矩陣、拒絕案例、Rules／索引部署核對 |
| 共用 CSS／design tokens | 全站視覺或功能耦合回歸 | 分段導入、受影響頁清單、響應式與功能回歸 |
| Hosting／CI | main 意外上線、Demo 外露 | PR 預覽、發布核准、ignore／路由核對、可回復版本 |
| 金流／EDM／Google／AI | 成本、個資、同意與外部副作用 | 明確授權、測試環境、最小資料、錯誤與取消策略 |

高風險不代表永久禁止修改；它代表修改前要有更清楚的契約，修改後要有更強的證據。

## 9. Git、預覽、發布與回復流程

### 9.1 建議標準流程

`確認工作包 → 建立獨立分支 → 實作與自我驗證 → PR／可閱讀差異 → 預覽環境 → 必要產品驗收 → 必要技術驗收 → 使用者核准發布 → 合併／部署 → 部署後驗證 → 完成交接`

現行機制下，**使用者核准發布的實際意義是「允許把已驗收版本合併到 `main`，而合併會立即觸發 live Hosting 部署」**。沒有「先合併、稍後再決定是否上線」的中間關卡。若未來改成手動 promotion 或受保護 environment，須更新本契約後才能採用不同時序。

### 9.2 發布閘門

- 不以「推到 main 後再看」作為正常預覽方式。
- 未取得使用者或其明確授權者的發布核准，不得合併到 `main`。
- HTML、CSS、JS、Rules 與設定檔必須保有可審查差異。
- 影響資料權限時，Hosting 通過不等於 Rules／索引已正確發布。
- 沒有預覽能力時，先回報限制與替代驗證；不可默認接受正式環境試錯。
- 正式發布前核對實際變更清單，排除 mockup、Demo、內部資料、暫存檔與秘密。
- 發布後至少驗證主要入口、登入、核心流程、權限拒絕與回復點。
- 已確認且仍可達的 P0，預設阻擋受影響的預覽與正式發布。若預覽共用正式 Firebase、Auth 或資料，亦視為可能影響正式環境；只有經證明隔離或不觸及缺陷路徑，才可排除於該次阻擋範圍。
- P0 例外發布必須由使用者逐項明確接受風險，並附工程與營運負責者提供的影響、緩解、監控與回復證據；代理不得自行默認例外。
- P1 不一律阻擋所有無關變更，但每次發布前必須逐項記錄：本次是否觸及、是否擴大暴露、緩解措施、負責者、期限及使用者是否接受殘留風險。觸及核心權限、資料隔離或無法回復的 P1，預設阻擋受影響發布。
- 缺陷摘要、詳細紀錄或實際程式狀態互相矛盾時，先標為「狀態待釐清」；受影響範圍在釐清前不得發布。

### 9.3 緊急修正

緊急修正仍需：

- 明確缺陷與影響範圍。
- 最小變更與指定寫入者。
- 至少一位相應領域審查者；若無法等待，需由使用者接受風險。
- 發布後補齊驗證、紀錄與必要的長期修正工作包。

## 12. 完成定義

一項工作只有在下列條件符合其風險等級時，才能標示為完成：

- 範圍內交付物已產出，範圍外檔案未被意外修改。
- 功能、內容與狀態符合已確認來源。
- 受影響的核心流程、資料、權限、響應式及可及性已驗證。
- PR／差異可閱讀，沒有秘密、個資、Demo 或未核准設定混入。
- 產品與技術驗收結果已分開記錄。
- 尚未驗證項目、風險與回復方式已揭露。
- 若已發布，正式環境已完成部署後驗證。

「已實作」不等於「已驗證」；「已驗證」不等於「已發布」；「已發布」也不代表品牌或產品決策永久定稿。

為避免使用者把「代理已交還檔案」理解成「現在已可驗收」，交付時至少分開標示：

| 維度 | 建議狀態 |
|---|---|
| 寫入權 | 未占用／占用中／已釋放 |
| 實作 | 未開始／進行中／實作完成 |
| Git | 未提交／已提交至非 main 分支／已推送／已開 PR |
| 驗證 | 未驗證／部分驗證／產品驗收通過／技術驗收通過／雙驗收通過 |
| 可見環境／發布 | 僅工作區／本機可看／預覽網址／正式站 |

- 不單獨使用「已完成」作為使用者交付結論；若使用，必須附上五個維度的限定。
- 寫入權「已釋放」只表示目前沒有寫入者，不代表已驗證、可預覽或可發布。
- 使用者驗收前應提供實際預覽網址；若沒有網址，第一段即明寫「目前無法由使用者查看」及尚缺哪一步。
- 代理可回報各維度的客觀狀態；整體工作包是否達成使用者目的並最終完成，由使用者確認。
