### 1.1 實作期間的技術修正

- **發布權杖改由 Cloud Function 持有**（原討論為前端持有 GitHub token）。原因：本專案 `functions/` 已存在並運作中（職務設計 AI 代理、授權碼與配額，Key 存於 Secret Manager），既有後端層讓「權杖不進瀏覽器」的安全做法幾乎零額外成本。**權杖任何時候都不得出現在前端。**
- **文章列表頁也改為 CI 產生的靜態頁**。原方案 B 讓列表從 Firestore 動態載入，目的只是避免手動維護清單；有了 CI 產生器後，靜態列表同時達成「自動維護」與「SEO 完整」，嚴格優於原方案。

| 欄位 | 內容 |
|---|---|
| **原內容** | `CLAUDE.md`：「Cloud-native, serverless, **no build step**」「Plain HTML + vanilla JavaScript…No framework, no bundler, no `package.json`, no transpile」 |
| **新決策** | 內容管線（`content/**` → `blog/**`、`sitemap.xml`、`feed.xml`）允許在 **CI 環境**執行 Node 產生器；**所有手寫應用頁**（`index.html`、`dashboard.html`、`tools/twa.html`、`tools/hti.html`、`report.html`、`timeline.html`、`admin.html` 等）維持純靜態、不進任何產生器 |

## 3. 內容模型

### 3.1 文章 frontmatter 欄位

```yaml
---
title: 文章標題                      # 必填，頁面 H1 與列表標題
seo_title: 搜尋結果用標題            # 選填，留白時等於 title
description: 一到兩句摘要            # 必填，meta description、OG、列表卡片共用
category: career                     # 必填，見 §3.2
tags: [工作價值, 轉職]                # 選填
cover: /assets/blog/xxx.jpg          # 選填，OG 圖與列表卡片；留白用品牌預設圖
date: 2026-07-24                     # 必填，發布日
updated: 2026-07-26                  # 選填，更新日
author: 何昆陽                        # 選填，預設品牌名
draft: false                         # 選填，true 則不產生、不進列表與 sitemap
---
```

### 3.2 分類（對齊既有工具三分類，不新增第四類）

| key | 對外名稱 | 範圍 |
|---|---|---|
| `career` | 職涯與工作 | 工作價值、興趣、能力、履歷定位、職缺分析、轉換選擇 |
| `life` | 生涯與角色 | 人生角色、時間與精力配置、關係、生活設計、長期調整 |
| `org` | 組織與人才 | 職務設計、人才標準、甄選流程、主管判斷、組織合作 |

分類為**單選且必填**；標籤為多選、自由詞彙。此三類與 `blog.html`、`tools.html` 現行分類骨架一致。

### 3.3 slug 規則

- 由後臺產生，可手動覆寫。
- 中文標題自動轉為 `YYYY-MM-DD-<簡短英數>` 或使用者自訂英數 slug。
- slug 一旦發布**不得變更**（會斷連結、掉 SEO 權重）；需更名時保留舊頁並加 canonical 指向新頁。

## 4. 路由與檔案結構

```
content/
  posts/<slug>.md          ← 作者內容（唯一真實來源，進 Git 版控）
  pages/<name>.md          ← 文案頁內容（如創辦人故事）
scripts/
  build-content.mjs        ← 產生器
blog.html                  ← 【產生物】文章列表（含分類篩選）
blog/<slug>.html           ← 【產生物】文章詳情
blog/category/<key>.html   ← 【產生物】分類頁
feed.xml                   ← 【產生物】RSS
sitemap.xml                ← 【產生物】全站 sitemap
robots.txt                 ← 手寫靜態
```

**產生物不進 Git**（加入 `.gitignore`），由 CI 在部署前產生。真實來源永遠是 `content/**` 的 Markdown。

> 例外：`blog.html` 目前是已存在的手寫骨架頁。實作時改為由產生器輸出，原骨架內容轉為產生器的模板；若無任何文章，輸出「內容建構中」的空狀態而非空白頁。

## 5. 產生器規格（`scripts/build-content.mjs`）

**輸入**：`content/posts/*.md`、`content/pages/*.md`
**輸出**：§4 所列產生物
**執行**：`node scripts/build-content.mjs`（本機可跑，CI 自動跑）
**相依**：`marked`（Markdown → HTML）。frontmatter 解析自行實作，減少相依面。

每篇文章頁必須產出：

1. 完整 `<head>`：`title`（用 `seo_title`）、`meta description`、`canonical`、Open Graph（`og:title/description/image/type=article/url`）、Twitter Card、JSON-LD `BlogPosting`（含 `datePublished`、`dateModified`、`author`、`image`）。
2. 沿用既有 `brand.css` 與站頭／站尾，與其他公開頁一致。
3. 麵包屑：首頁／文章／分類／本篇。
4. **右側目錄（TOC）**：由 H2／H3 自動產生，H2/H3 自動加 `id`，捲動時高亮當前段落；行動版收合。
5. 文章底部：分類與標籤、**電子報訂閱區塊**、社群連結、相關文章（同分類最新 3 篇）。
6. `draft: true` 一律跳過。

**失敗行為**：任何必填欄位缺漏、slug 重複、Markdown 解析失敗 → 產生器以非零狀態結束 → **CI 中止、不部署**、線上維持前一版，並在 Actions 記錄明確錯誤訊息。

## 6. SEO 交付清單

### 6.1 現況（2026-07-24 實測）

全站 **零** Open Graph 標籤、**無** canonical、**無** sitemap.xml、**無** robots.txt、**無** 結構化資料。僅部分頁面有基本 `meta description`。

### 6.2 交付項目

| 項目 | 範圍 | 批次 |
|---|---|---|
| `canonical` | 所有公開頁 | 1 |
| Open Graph（title/description/image/url/type/site_name/locale） | 所有公開頁 | 1 |
| Twitter Card（`summary_large_image`） | 所有公開頁 | 1 |
| JSON-LD `Organization` + `WebSite` | 首頁 | 1 |
| `robots.txt` | 站根，禁爬 `dashboard/admin/counselors/report/timeline` 等私有頁 | 1 |
| `sitemap.xml` | 先手寫涵蓋現有公開頁，批次 2 起改由產生器輸出 | 1 → 2 |
| JSON-LD `BlogPosting` | 每篇文章 | 2 |
| `feed.xml`（RSS） | 全站文章 | 2 |
| 分類頁 canonical 與分頁處理 | 分類頁 | 2 |

**預設 OG 圖**：需要一張 1200×630 品牌圖。尚未提供前，先以現有 `ormap-logo.png` 為 fallback，並列為待補項目。

**私有頁一律加 `noindex`**：`dashboard.html`、`admin.html`、`counselors.html`、`report.html`、`timeline.html`、`whiteboard.html` 及後臺頁。

## 7. 發布後臺（批次 3）

### 7.1 作者體驗

1. 登入後臺 →「新增文章」。
2. 上半部欄位：標題、SEO 標題、分類、標籤、摘要、封面、發布日、slug。
3. 下半部大文字框：**從 Notion／Obsidian 直接貼上 Markdown**。
4. 即時預覽（排版與 TOC）。
5. 按「發布」或「存為草稿」。
6. 約一至兩分鐘後文章自動上線。

### 7.2 技術路徑

```
後臺頁（Firebase Auth + isSuperAdmin 檢查）
   ↓ onCall
Cloud Function publishPost
   ↓ GitHub Contents API（權杖存 Secret Manager，最小權限：僅本 repo contents write）
commit content/posts/<slug>.md 到 main
   ↓
GitHub Actions：npm ci → node scripts/build-content.mjs → 部署 Hosting
```

### 7.3 安全與權限要求

- 權杖**只存在 Secret Manager**，不進前端、不進 Git、不寫入 log。
- Function 內必須**重新驗證** caller 的 `isSuperAdmin`，不信任前端傳入的身分。
- 權杖使用 GitHub fine-grained PAT，權限鎖定**單一 repo 的 Contents 寫入**。
- 輸入需消毒：slug 僅允許 `[a-z0-9-]`，防止路徑穿越（`../`）。
- 圖片上傳走 **Firebase Storage**，後臺取得公開 URL 後插入 Markdown；Storage 規則須限制僅管理者可寫。

> **注意**：發布會直接 commit 到 `main`，而推送 `main` 即觸發 live Hosting 部署。這代表**後臺發布＝正式上線**。此行為必須在後臺 UI 明確告知，且此設計需使用者最終確認（見 §10）。

## 8. 電子報與社群串聯（批次 4）

- 供應商：第三方 EDM，**待選定**。建議條件：有免費方案、支援繁體中文、**支援 RSS-to-Email**（新文章自動轉為電子報，與 §6 的 `feed.xml` 直接串接）、有合規的雙重確認與退訂機制。
- 訂閱區塊為**共用元件**，出現在：文章底部、`blog.html`、`index.html`、`about.html`。
- 文案框架綁定「見證藍圖成形」概念（見 §9）。
- 同時列出主理人與夥伴的社群入口（IG 等）。
- **治理紅線**：在供應商確定並實際串接完成前，**不得放上外觀可按但無作用的訂閱表單**（AGENTS.md §7 禁止把未實作功能呈現為可操作）。批次 4 需求未滿足時，該區塊不上線。
- 需一併提供：蒐集目的、最小欄位（僅 email）、同意聲明、退訂方式，並更新 `privacy.html`。

## 9. 「建構中／見證藍圖成形」呈現規則

- 沿用既有三態徽章：`site-status--live`（可以用）、`site-status--build`（還在蓋）、`site-status--plan`（規劃中）。
- 定義一段全站共用的說明文案，放在骨架頁與訂閱區塊，誠實說明目前進度與邀請見證的邀約。
- **不得**用此概念包裝不存在的功能；徽章標示的是實際狀態，不是行銷語言。
- 正式文案由使用者定稿；實作先提供結構與草稿。
