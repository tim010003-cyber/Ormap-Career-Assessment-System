## 1. 文件目的

本文件定義 ORMAP 從目前的「公開首頁＋兩項評測＋個人 Dashboard＋諮詢後台」逐步轉型為「內容、評測、課程與服務相互承接的平台」時，應採用的產品優先序、網站架構、遷移路徑與責任邊界。

本文件重點不是一次完成所有理想功能，而是確保每一階段：

1. 有明確的使用者問題與產品成果。
2. 延續現有可用功能，不為了改版破壞評測、報告、白板或權限。
3. 不讓內容、評測、課程與諮詢各自形成孤島。
4. 不把規劃中、Demo 或未驗證功能描述為已上線。
5. 讓品牌決策、內容定義與 Cloud 工程實作使用相同基準。

### 1.1 決策優先序

若文件內容互相衝突，依下列順序處理：

1. 使用者最新、明確的產品與品牌決策。
2. `原味藍圖_品牌與網站架構總綱.md` 的已定稿／已確認內容。
3. 本文件的階段策略與目標架構。
4. `PRD/Ormap_系統規格書_v4.0_2026-07-16.md` 的產品方向。
5. `PRD/Ormap_系統規格書_v3.8_2026-07-16.md` 的既有技術現況。
6. mockup、Demo、註解或未完成原型。

若「已定稿」來源彼此仍有衝突，不得自行混寫；需列入待使用者決策。

## 2. 最新產品方向

### 2.1 長期平台關係

原味藍圖長期將形成以下服務漏斗：

```text
文章／影片建立理解與信任
→ 免費評測提供初步體驗
→ 進階評測提供更完整的分析
→ 實體課／直播課協助學習與應用
→ 個人或企業諮詢承接高度個別化問題
→ Dashboard 保存資料與長期關係
→ EDM、內容與新產品維持後續互動
```

這不是強迫所有使用者走相同線性路徑。文章、評測、課程、諮詢與組織方案均可成為入口，但每個入口都必須提供合理的下一步。

### 2.2 已確認的開發優先序

```text
第一階段：評測工具
→ 第二階段：品牌網站、文章與內容後台
→ 第三階段：實體／直播課程與外部諮詢承接
```

第一階段以「職務設計系統」作為第一個重點模組，原因是評測與結構化工具屬於 ORMAP 的獨家 Know-how，市場替代方案相對少；課程與一般內容服務已有較多外部替代方案，因此延後開發。

### 2.3 已確認的產品決策

#### 評測

- 第一階段的開發主體是評測工具平台。
- 第一個重點模組是職務設計系統。
- 免費與付費版本主要以「評測完整度」形成差異。
- 簡化版本可能只開放前段功能。
- 題目深度、報告差異、免費範圍、付費範圍與定價仍待個別工具規格確認。

#### 課程

- 課程安排於第三階段。
- 主要形式是實體課與直播課，不以預錄自學課為主要產品。
- 課程必須登入後才能報名。
- 帳號資料用於取得姓名／顯示名稱與 Email，供報名和通知使用。
- 若未來需要真實姓名、公司、職稱或其他資料，需另設報名欄位、用途說明與隱私告知；Google 顯示名稱不等於經驗證的法定姓名。
- 課程最終需支援免費／付費梯次、金流、Email 通知與 Google Calendar。

#### 諮詢

- 不優先開發站內諮詢預約與管理系統。
- 第一階段保留 LINE 作為主要聯絡入口。
- 個人客戶（to C）延續內嵌 Notion 表單與外部承接。
- 個人諮詢目前規劃基準：約 90 分鐘、2,500 元。
- 企業／組織客戶（to B）依需求另行報價；目前規劃基準約 2 小時、6,000 元。
- 價格正式發布前仍需確認服務內容、付款方式、取消／改期、稅務與資料處理說明。

#### Blog 與內容

- 第二階段建立 Blog 與內容管理後台。
- 支援文章發布。
- 未來可內嵌 YouTube 影片。
- 文章與影片需能連到相關評測、課程、訂閱或其他下一步。

#### Index 與 Dashboard

- Index 是永久公開的品牌網站、內容與產品入口。
- Dashboard 是登入後個人控制台。
- 兩者必須分開。
- 已登入使用者仍應能閱讀 Index、文章、課程與公開工具介紹，不應被首頁自動強制轉址。
- Dashboard 應作為摘要與分流，不再同時塞滿全部工具與全部歷史報告。

## 3. 現況基準

### 3.1 現有正式功能

| 模組 | 現況 |
|---|---|
| 公開首頁 | `index.html` 已移植新版吸引子視覺與品牌內容，但登入者會自動轉至 dashboard。 |
| 個人中心 | `dashboard.html` 同時呈現工具入口與歷史紀錄。 |
| TWA | `tools/twa.html` 可評測、儲存草稿、寫入 Firestore、產生報告資料並支援白板鏡像。 |
| HTI | `tools/hti.html` 可評測、保存本機進度、寫入 Firestore、呈現頁內結果並支援白板鏡像。 |
| 歷史報告 | `report.html?id=...` 讀取 TWA／HTI assessments。 |
| 成長軌跡 | `timeline.html` 在至少兩筆 TWA 時由 dashboard 顯示入口。 |
| 諮詢白板 | `whiteboard.html`、TWA／HTI mirror mode、即時 Firestore 同步。 |
| 管理後台 | `admin.html`、`counselors.html` 提供個案、報告、白板與諮詢師管理。 |
| 回饋與法務 | `feedback.html`、`privacy.html`、`terms.html`。 |

### 3.2 職務設計 Demo 現況

`tools/job-design/` 已有可端到端操作的前端 Demo，但尚非正式產品：

- 使用 localStorage 作為主要持久層。
- 使用規則式／樣板式 Mock AI，不呼叫正式 AI 後端。
- 學員／講師身分可在畫面自由切換，非 Firebase Auth 權限。
- 可建立案例、逐模組輸入與確認、產生 Word。
- 沒有正式會員權益、後端安全、組織資料隔離或正式產品入口。
- 正式首頁的職務設計卡尚無 CTA，Demo 也未接入正式導覽。

Demo 可作為互動、資料欄位和使用流程的驗證來源，但不得直接宣稱已可正式使用。

### 3.3 現有技術架構

- 靜態 HTML＋Vanilla JavaScript ES modules。
- 無 bundler、框架或 build step。
- Firebase Hosting 發布 repo 根目錄。
- Firebase Google Authentication。
- Cloud Firestore。
- TWA／HTI 以本地 JSON 驅動內容。
- Tailwind CDN、Lucide、Google Fonts，部分頁面使用 Chart.js。
- mockups 與 job-design Demo 目前未被 Hosting 明確排除。

## 6. 現況與目標 Gap

| 面向 | 現況 | 目標 | 遷移方向 |
|---|---|---|---|
| Index | 新品牌首頁；已登入者自動轉 dashboard | 永久公開的品牌與內容入口 | 取消強制轉址；登入只改變導覽狀態 |
| 開場體驗 | 無開場封面 | 可跳過、可降級的「打開」體驗 | 同頁開場層，不建立阻擋式 Splash |
| 導覽 | 每頁各自維護 | 一致的公開導覽與會員導覽 | 先定義命名、入口與共享責任 |
| Dashboard | 工具與歷史報告混合 | 個人摘要與分流 | 拆出「我的探索」「我的報告」 |
| 工具目錄 | 首頁與 dashboard 各一套 | 單一正式工具中心 | 集中工具名稱、狀態與入口 |
| TWA／HTI | 可運作但名稱、UI、流程各自獨立 | 共用評測外殼與一致命名 | 保留核心計分／資料，逐步整合外層 |
| 職務設計 | 孤立 localStorage／Mock AI Demo | 正式帳號、資料、權限與安全後端 | 以垂直切片產品化，不直接發布 Demo |
| 免費／付費 | 無共同商務模型 | 依評測完整度提供版本 | 先建版本／權益概念，金流後接 |
| Blog | 無正式內容系統 | 有文章前台與發布後台 | 第二階段建立內容模型與 CMS |
| EDM | 無 | 訂閱、同意、退訂與新內容通知 | Blog 基礎完成後建立 |
| 課程 | 無正式頁／報名 | 實體／直播課程與登入報名 | 第三階段建立輕量課程系統 |
| 諮詢 | 報告導向 LINE | 正式服務說明＋LINE／Notion | 不建站內排程，補完整承接頁 |
| 組織端 | 首頁卡片無 CTA | 獨立組織方案與洽詢 | 由職務設計系統建立第一條正式路徑 |
| 後台 | CRM／諮詢師管理 | 內容、產品、課程、營運分區 | 共用權限，不塞進單一頁面 |
| mockup／Demo | 可能由 Hosting 直接提供 | 正式與測試環境明確隔離 | 後續決定排除或建立受控測試環境 |

## 7. 分階段遷移計畫

### 第 0 階段：平台基礎與決策對齊

目的：讓後續功能不再因路由、名稱、內容來源與登入流程反覆重做。

交付：

1. 正式頁面地圖與路由映射。
2. 公開站與會員站責任邊界。
3. 工具正式名稱映射。
4. 頁面／區塊穩定 ID。
5. 單一內容來源模型。
6. 登入後返回原目標的流程。
7. 工具成熟度與免費／付費／Demo 狀態模型。
8. 現有 TWA、HTI、報告、timeline、白板回歸清單。

完成標準：

- 使用者與工程端能精確描述「哪一頁、哪一區、哪個欄位」。
- 後續任務不需自行猜測工具名稱、上線狀態或 CTA。
- 現有核心功能有明確的不可退化清單。

### 第一階段：評測工具平台

#### 1A. 分開 Index 與 Dashboard

- Index 保持公開。
- 已登入者不再被自動送走。
- Header 依狀態顯示「登入」或「個人空間」。
- 建立一致的公開工具入口。
- 登入完成後返回原工具。
- Dashboard 摘要化。
- 建立「我的探索」與「我的報告」資訊層。

#### 1B. 職務設計系統產品化

1. 確認正式 MVP 範圍與首批使用者。
2. 定義免費版與進階版的完整度邊界，但可延後接金流。
3. 將假身分切換改為 Firebase Auth 與真實角色。
4. 將 localStorage 唯一持久層改為正式資料保存；localStorage 可保留草稿用途。
5. 保留 Mock AI 可替換介面；正式 AI 經安全後端呼叫。
6. 保存原始輸入、系統整理、待確認推論、人工確認與輸出版本。
7. 支援案例列表、進度恢復、保存、下載與 Dashboard 返回。
8. 建立公開職務設計介紹與真實 CTA。

#### 1C. 整合 TWA／HTI

- 建立公開工具介紹。
- 統一登入與返回工具流程。
- 共用導覽、進度、完成與報告出口。
- 統一版本／權益／狀態標籤。
- 統一正式名稱，但名稱變更需使用者決策。
- 保留計分、資料 JSON、草稿、歷史紀錄與 mirror mode。

第一階段完成標準：

- 使用者能從公開網站理解工具，登入後回到指定工具。
- 職務設計有一條正式可完成、可保存、權限正確的垂直流程。
- Dashboard、我的探索、我的報告職責分開。
- TWA／HTI 現有核心功能沒有退化。
- Demo、規劃中與正式可用功能不會混淆。

### 第二階段：網站內容與 Blog

交付：

1. 文章／影片內容模型。
2. 文章列表與內容頁。
3. YouTube 內嵌。
4. 分類與標籤。
5. 相關文章與相關工具推薦。
6. 內容管理後台。
7. 草稿、預覽、發布、下架。
8. SEO 基本欄位。
9. EDM 訂閱、同意、退訂與新內容通知。

內容來源原則：

- 核心品牌與工具定義：版本控制、需審核。
- 經常發布的文章與營運內容：由內容後台管理。
- 使用者版內容表與工程映射必須來自同一來源。

### 第三階段：課程與諮詢

#### 課程 MVP

- 實體課與直播課。
- 課程列表與詳情。
- 梯次、日期、地點／直播方式、名額。
- 登入後報名。
- 報名姓名／Email 與必要資料告知。
- 免費／付費狀態。
- 金流。
- Email 確認與提醒。
- Google Calendar。
- Dashboard「我的課程」。

非首期範圍：

- 大型預錄 LMS。
- 複雜單元進度。
- 證書、討論區與不符合目前授課模式的功能。

#### 諮詢承接

- 個人諮詢服務頁。
- 組織諮詢服務頁。
- 服務內容、適合對象、時間與目前方案基準。
- LINE／Notion 表單。
- 隱私、取消／改期與必要服務說明。

非首期範圍：站內排程、諮詢 CRM、複雜時段管理與自動改期。

## 8. 職務設計系統正式 MVP 規劃框架

### 8.1 必須先確認

- 首批使用者：個人學員、主管／HR、企業客戶或封閉測試。
- 第一版包含哪些模組與輸出。
- 免費範圍與進階完整度。
- 是否先以 Mock AI 驗證，何時接正式 AI。
- 案例保存期限、刪除、下載與重新開放規則。
- 個人案例與組織案例的資料隔離。

### 8.2 正式資料責任

至少包含：

- 使用者與角色。
- 職務案例。
- 所屬個人／組織。
- 模組進度。
- 原始輸入。
- 系統整理結果。
- 待確認推論與缺口。
- 人工確認結果。
- 輸出版本。
- 下載紀錄。
- 版本／權益狀態。
- 建立、更新與必要稽核時間。

### 8.3 AI 安全層

```text
前端
→ Firebase／Cloud 安全後端
→ 驗證帳號、角色與版本權益
→ 組裝指定任務
→ 呼叫 AI
→ 保存任務版本與可追溯結果
→ 回傳前端
→ 使用者人工確認
```

不可將 API key、付費呼叫、授權判斷或秘密放入前端。AI 只整理、提問或提出待確認推論，不得替使用者決定職務是否成立、最低標準或正式發布內容。

### 8.4 完成出口

使用者完成後至少能：

- 保存案例。
- 查看完成進度。
- 回到個人 Dashboard。
- 下載已確認輸出。
- 查看輸出版本。
- 建立下一個案例。
- 理解可用的進階版本、課程或企業諮詢下一步。

## 10. Cloud 技術演進方向

本節為高階方向，不授權立即建立雲端資源或外部服務。

### 10.1 可延續

- Firebase Hosting：公開網站與靜態資源。
- Firebase Auth：Google 登入與會員識別。
- Firestore：評測、案例、報告、內容 metadata、報名與權益等結構化資料。
- 現有 TWA／HTI JSON 資料管線。

### 10.2 後續可能需要

- Cloud Functions 或 Cloud Run：安全 AI、金流 webhook、Email、Calendar、權益驗證。
- Cloud Storage：文章、課程或輸出附件。
- 排程／任務機制：課程提醒與營運通知。
- 正式 Email／EDM 供應商：交易通知與行銷訂閱分開處理。
- 金流供應商：共用商品、訂單、付款與權益層，不為每項產品重做付款。

所有付費 API、外部發送、金流與 Cloud 資源必須另經使用者核准。

## 11. 專案運作方式


### 11.2 開發後驗證

- 公開入口與返回路徑。
- 未登入／已登入流程。
- 手機、平板與桌機。
- 鍵盤、focus、語意標題、對比與替代文字。
- 儲存、恢復、完成與錯誤狀態。
- 報告與下一步。
- 權限與資料隔離。
- 既有 TWA／HTI／report／timeline／whiteboard 回歸。
- 工具狀態與功能真實一致。

- 跨頁命名、CTA、狀態、響應式與可及性。
- 整合驗收與 Gap 管理。

### Claude 協作

- 在已確認架構與介面契約下，維護既有 TWA／HTI 評測核心。
- 保護計分、JSON 資料、草稿、FireStore 儲存與 mirror mode。

- 不自行決定品牌定位、工具名稱、上線狀態、主要 CTA、免費／付費範圍或全站 IA。

### 協作規則

- 不同代理不得同時修改同一檔案。
- 全站架構、路由、共用內容和名稱由單一主責維護。
- 模組交付前先定義輸入、輸出、資料、權限與驗收契約。
