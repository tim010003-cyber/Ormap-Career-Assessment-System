## 1. 本輪新發現（不在原登記簿內）

INTF-001 的逐檔核對過程中，發現四項既有登記未涵蓋的缺陷。其中 SEC-005 改變了整體嚴重度判斷，**應優先於原 SEC-001～004 處理**。

### SEC-005｜白板結束流程可被用於取得他人評測報告的存取權（權限提升鏈）

- **狀態**：已確認／未修復
- **嚴重度**：**P0 緊急**（唯一一項需即刻處置者）
- **確認依據**：
  - `firestore.rules:63-65`：`whiteboard_sessions` 的 read＋write 只要求已登入。
  - `whiteboard.html:320`：角色判定 `userRole = sessionData.counselorUid === currentUser.uid ? 'counselor' : 'student'` 為**純客戶端**判定，伺服端無對應驗證。
  - `whiteboard.html:552-573`：`confirmEnd()` 以 `where('uid','==',studentUid)` 取出該 uid 的所有 `assessments`，並批次寫入 `assignedCounselorUid = 當前登入者`、`counselorAccess = true`。
  - `firestore.rules:42-45`：`assessments` 的 update 只要求 `isCounselor()`，未驗證該諮詢師與該筆資料的關係。
- **形成鏈條**：session 文件的 `studentUid` 欄位可被任意登入者改寫（依 rules:63），而 `confirmEnd()` 完全信任該欄位來決定要綁定誰的報告。兩者相加，使「宣告某人是我的學員」等同「取得該人全部評測報告的存取權」。
- **可能影響**：任意登入者可取得任意使用者全部 TWA／HTI 報告的持續讀取權，且該授權會寫入資料（`counselorAccess: true`），在 Rules 修好後仍**殘留於既有資料**。
- **期望行為**：`assignedCounselorUid` 的寫入必須由伺服端條件約束——寫入者須為合法諮詢師，且須存在一筆 `counselorUid == 寫入者 && studentUid == 目標` 的 session。
- **關鍵注意**：本項需要**資料清查與回溯**，不是只改 Rules 就結束。修復後仍須盤點既有 `assignedCounselorUid` 是否有非預期綁定。
- **驗收條件**：非參與者的 session 欄位寫入被拒；`assessments.assignedCounselorUid` 的越權寫入被拒；既有資料完成盤查；正常結束流程回歸通過。

### SEC-006｜白板學員歡迎頁存在反射型 XSS

- **狀態**：已確認／未修復
- **嚴重度**：P1 高
- **確認依據**：`whiteboard.html:623-631` 於 `getDoc` 失敗或文件不存在時（catch 為空，`catch(e){}`），`counselorName` 與 `tool` 回退至 URL 參數 `?c=` 與 `?t=`；`whiteboard.html:646-711` 以樣板字串直接插入 DOM，未經跳脫，且同時進入 HTML 屬性語境（654）與文字語境（655）。
- **可能影響**：透過構造連結在本站 origin 下執行腳本。因白板頁與 Firebase Auth 同源，可觸及已登入者的憑證環境。
- **期望行為**：所有來自 URL 或 Firestore 的字串以 `textContent` 或跳脫後插入；`getDoc` 失敗需顯式處理而非靜默回退。
- **關聯**：`report.html:509` 的 HTI 分支同樣未跳脫 `userName`（TWA 分支 `report.html:812` 有 `escHtml`，兩分支不對稱）；`timeline.html:217-229` 的 `focusType` 亦直接內插。三處應一併處理。

### SEC-007｜HTI 草稿未綁定使用者，共用裝置會跨帳號外洩並錯誤歸戶

- **狀態**：已確認／未修復
- **嚴重度**：P1 高
- **確認依據**：`tools/hti.html:665` 以固定 key `hti_progress_v4` 寫入 localStorage，**不含 uid**；`tools/hti.html:675-677` 的 `restoreProgress()` 無條件提示還原。對照 TWA 走 `user_progress/{uid}_TWA`（`tools/twa.html:2692`）具天然隔離。

### OPS-002｜白板即時同步的寫入頻率過高

- **嚴重度**：P2 中

### 2.1 SEC-001 的範圍比原登記小

原登記寫「目前 HTI 主要使用瀏覽器 localStorage，是否另有雲端草稿相依尚未驗證」。**已驗證：HTI 完全不使用 Firestore 草稿。**

- 全檔搜尋 `user_progress` 於 `tools/hti.html` 零命中；僅 `tools/twa.html` 使用（2425、2680、2692、2729）。
- 因此 `user_progress` 的 Rules 收緊**只影響 TWA**，回歸範圍可縮小。
- 文件 ID 格式確認為 `user_progress/{uid}_TWA`，內容為 `{ data: <localStorage 原始 JSON 字串>, updatedAt: serverTimestamp() }`。
- 但 HTI 另有 SEC-007 的獨立問題，兩者不可視為同一件事。

### 2.2 SEC-002 應升為 P0，且需與 SEC-005 合併處理

原登記標 P1 並註「若正式資料含敏感諮詢內容且已可被非參與者存取，需重新評估是否升為 P0」。**該條件成立**：`whiteboard_sessions` 文件直接含 `studentName`、`studentEmail`、`counselorEmail`（`whiteboard.html:289`、`742-744`），且 SEC-005 顯示其可寫性會外溢成 `assessments` 的存取權。

SEC-002 與 SEC-005 是**同一個根因**（session 無參與者驗證）的兩個表現，應放在同一個工作包，不可分開發布。

### 3.1 參與者模型的三個陷阱

撰寫 Rules 時，以下三點若忽略會直接讓正常流程壞掉：

1. **學員在成為 `studentUid` 之前，必須already能 `get` 該 session。** `whiteboard.html:625` 的 `showStudentWelcome()` 需先讀取文件才能顯示諮詢師姓名與工具。因此不能寫成「只有 `counselorUid`／`studentUid` 相符才可讀」。
2. **`studentUid` 的首次寫入是一個「認領」動作。** 寫入當下 `resource.data.studentUid` 為 `null`，`request.auth.uid` 尚未在文件中。Rules 需允許 `resource.data.studentUid == null && request.resource.data.studentUid == request.auth.uid` 這一種轉換，且**只能發生一次**（現行 `whiteboard.html:739` 的先到先得檢查讀的是快取值，存在競態，見 §5）。
3. **諮詢師會刪除學員的 `whiteboard_active/{studentUid}`**（`whiteboard.html:546`），此時 doc id ≠ `request.auth.uid`。若 Rules 寫成 `allow write: if request.auth.uid == uid`，結束流程會被拒絕，且 `whiteboard.html:547` 的 `Promise.all` 會連帶跳過 `assessments` 綁定（見 §5 B11）。

**目前沒有任何憑證機制**：session 連結本身即唯一憑證（bearer link），無邀請碼、無名單。這是產品決策，非缺陷，但 Rules 設計必須明確承認此前提——否則會設計出與現行流程不相容的規則。**此點需使用者確認是否維持。**

## 5. 修復時必須一併處理的既有耦合

以下是修 Rules 時會踩到的既有行為。若不處理，Rules 正確但功能會壞：

- **`whiteboard.html:547` 的 `Promise.all`**：`deleteDoc` 若因權限被拒即整條 `.then` 跳過，`assessments` 綁定不執行，且使用者看不到錯誤、頁面不跳轉（B11）。收緊 Rules 前必須先改錯誤處理。
- **`whiteboard.html:739` 的學員認領競態**：讀的是 `showStudentWelcome` 載入時快取的 `_pendingSession.sessionData`（716），非即時值。兩人同時開連結會互相覆寫。應改用 transaction 或以 Rules 的 create-only 語意約束。
- **`report.html:346` 的空 `catch(e){}`**：吞掉 `counselors/{uid}` 的 permission-denied，使「Rules 設錯」與「真的無權限」外觀完全相同。收緊 `counselors` 讀取權前必須先修此處，否則除錯成本極高。
- **`timeline.html:151-154` 的靜默失敗**：查詢失敗只 `console.error` 並隱藏 loading，畫面全空白。缺索引時 Firestore 正是丟 failed-precondition——這正是 OPS-001 在生產環境的實際症狀。
- **`assessments` 的 `list` 收緊會同時影響**：`dashboard.html:200`、`timeline.html:119-126`、`admin.html:299-301`、`whiteboard.html:552-555`。四處查詢形狀必須與 Rules 對齊，缺一即整批失敗。

## 6. 索引需求（OPS-001 的具體內容）

> **⚠️ 本節於 2026-07-20 修正。** 初版依靜態推論宣稱「只需要一個 composite index，dashboard 不需要」，該判斷**錯誤**。使用者授權後以 `firebase firestore:indexes` 查得實際狀態，更正如下。Firestore 在「等值過濾欄位」與「排序欄位」不同時即需要 composite index，`dashboard.html` 正屬此情形。

### 6.1 Cloud 專案中實際存在的 composite indexes（已查證）

查詢方式：`firebase firestore:indexes --project careervalue-ormap`（唯讀，未修改任何設定）。共 4 個，全部在 `assessments`：

| # | 欄位序 | 對應查詢 | 判定 |
|---|---|---|---|
| 1 | `assignedCounselorUid` ASC, `createdAt` ASC | 無現行查詢完全對應（`admin.html:300-301` 只有等值、無排序） | **疑似歷史遺留**，待確認 |
| 2 | `counselorAccess` ASC, `createdAt` DESC | 無任何現行查詢使用 | **疑似未使用**，待確認 |
| 3 | `toolType` ASC, `uid` ASC, `createdAt` DESC | `timeline.html:119-126` | **使用中，必要** |
| 4 | `uid` ASC, `createdAt` DESC | `dashboard.html:200-202` | **使用中，必要** |

`fieldOverrides`：0 個。

### 6.2 現行查詢與索引的對應

- `timeline.html:119-126`（`uid ==` + `toolType ==` + `orderBy createdAt desc`）→ 索引 3。
- `dashboard.html:200-202`（`uid ==` + `orderBy createdAt desc`）→ 索引 4。
- `admin.html:299`（Super Admin：只有 `orderBy createdAt desc`）→ 單欄位索引即可。
- `admin.html:300-301`（一般諮詢師：只有 `assignedCounselorUid ==`）→ 單欄位索引即可。
- `whiteboard.html:552-555`（只有 `uid ==`）→ 單欄位索引即可。

### 6.3 OPS-FIX-G 的具體內容

1. 由上述實際狀態產生 `firestore.indexes.json` 納入版控（**至少含索引 3、4**；索引 1、2 先保留並標註待確認，不要在未確認前刪除——刪除索引會使依賴它的查詢立即失敗且重建耗時）。
2. 於 `firebase.json` 的 `firestore` 區塊加入 `"indexes": "firestore.indexes.json"`。
3. 補上部署與回復方式的紀錄。

**排序注意**：SEC-FIX-F 若把 `list` 改為伺服端過濾（例如加上 `where('assignedCounselorUid','==',...)` 或 `counselorAccess`），**會產生新的 composite index 需求**。因此 OPS-FIX-G 的「定案」不應早於 SEC-FIX-F，否則要建兩次；但「先把現況納入版控」可以立即做，風險為零。

## 6A. 資料盤查的執行限制（2026-07-20，使用者已授權唯讀查詢）

- 以 `firebase-admin` 撰寫腳本需要服務帳戶金鑰。**金鑰屬秘密憑證，不應交由代理處理**，故未採用。

**方式 A — Firebase Console（最省事）**：Firestore Database → `assessments` → 以 `assignedCounselorUid` 篩選，比對每筆的 `assignedCounselorUid` 是否確實為該學員的實際諮詢師。重點是找出「`counselorAccess` 為 true 但雙方從未有過 session」的紀錄。

1. **是否授權進行資料盤查**（唯讀查詢正式 Firestore，用於量化 SEC-005 與 AUTH-001 的影響範圍）。這是決定修復方式的前提。
2. **白板的 bearer link 模型是否維持**（見 §3.1）。若要加入邀請碼或名單，Rules 設計與工作量會顯著不同。
3. **SEC-003：評測建立是否仍需支援未登入**。目前 Rules 允許匿名建立，但 UI 流程實際依賴登入 uid。
4. **是否同意先行發布 SEC-FIX-B 與 SEC-FIX-C**（純前端、不動 Rules、可獨立回復）。
5. **白板標注功能的去留**（見 INTF-001 報告 §附錄）：整套 canvas 標注系統無任何 UI 可觸發，約佔 `whiteboard.html` JS 的 15%。這影響保護清單要不要保護它。
