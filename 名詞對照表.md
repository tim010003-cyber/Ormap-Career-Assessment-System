## 一、每天都會遇到的（最重要）

| 你會看到的詞 | 中文說法 | 白話解釋 |
|---|---|---|
| **PR**（Pull Request） | **合併申請** | 「我改好一批東西，請你看過再決定要不要放進正式版」。像交作業等審核，不是直接改。 |
| **branch**（分支） | **草稿線** | 從正式版複製一份出來改，改壞了也不影響正式網站。改好才申請合併。 |
| **merge**（合併） | **併入正式版** | 把草稿線收進正式版。⚠️ **這個專案一合併就等於上線**，中間沒有第二道關卡。 |
| **commit** | **存檔紀錄** | 把一批改動封存成一個紀錄點，附上說明。沒 commit 的東西很脆弱，可能被誤刪。 |
| **deploy**（部署） | **上線／發布** | 把檔案送到網路上讓大家看得到。 |
| **preview channel** | **試看網址** | 臨時網址，讓你先看實際畫面再決定要不要上線。7 天後自動失效。 |
| **rollback** | **回復上一版** | 上線後發現有問題，退回前一個版本。 |
| **repo / repository** | **程式碼倉庫** | 存放整個專案所有檔案的地方（在 GitHub 上）。 |
| **main** | **正式版主線** | 專案的正式分支。推上去就會自動上線。 |

## 二、設計與開發階段（Codex 那邊常用）

| 你會看到的詞 | 中文說法 | 白話解釋 |
|---|---|---|
| **mockup** | **設計稿／示意稿** | 只有外觀、不能真的用的假頁面。像裝潢示意圖——好看但沒水電。 |
| **prototype** | **試作品／原型** | 比設計稿再進一步，可以點、可以操作，但資料是假的。 |
| **demo** | **展示版** | 給人看效果用的，不是正式功能。 |
| **wireframe** | **線框圖** | 更早期的草圖，只有方塊和線，標示東西放哪裡。 |
| **IA**（Information Architecture） | **資訊架構** | 網站有哪些頁、怎麼分類、從哪裡點到哪裡。 |
| **UI** | **使用者介面** | 看得到的東西：按鈕、顏色、排版。 |
| **UX** | **使用者體驗** | 用起來順不順、找不找得到、會不會卡住。 |
| **design tokens** | **設計變數** | 把顏色、字級、間距集中定義成一份清單，全站共用。改一處全站跟著變。 |
| **responsive** | **響應式** | 同一個網頁在電腦、平板、手機上都能正常顯示。 |
| **accessibility / a11y** | **無障礙** | 讓視障者、只能用鍵盤的人也能操作。 |
| **CTA**（Call To Action） | **行動呼籲** | 那些要人點的按鈕文字，例如「開始評測」「立即報名」。 |
| **hero** | **首屏主視覺** | 網站最上面那一大塊，進站第一眼看到的標題與主圖。 |

## 三、後端與資料（Claude 那邊常用）

| 你會看到的詞 | 中文說法 | 白話解釋 |
|---|---|---|
| **Firebase** | （產品名，不翻） | Google 提供的一整套雲端服務，我們的網站、登入、資料庫都在上面。 |
| **Firestore** | **雲端資料庫** | 存學員作答、報告、諮詢紀錄的地方。 |
| **collection / document** | **集合／文件** | 資料庫的分層方式。集合像資料夾（例如「所有評測結果」），文件像裡面的一筆資料（某人的某次作答）。 |
| **Firestore Rules** | **資料庫門禁規則** | 決定「誰可以看誰的資料」。這是我們資安問題的核心。 |
| **index**（Firestore index） | **資料庫索引** | 像書的目錄。沒有它，某些查詢會直接失敗、畫面空白。 |
| **Auth**（Authentication） | **登入驗證** | Google 登入那套。 |
| **uid** | **使用者編號** | 每個登入者的唯一身分編號，資料靠它區分是誰的。 |
| **query** | **查詢** | 去資料庫撈資料的動作，例如「撈出這個人所有的評測」。 |
| **Cloud Functions** | **雲端函式** | 跑在伺服器上的小程式。需要保密的事（金流、AI 金鑰）必須放這裡，不能放在網頁裡。 |
| **API key / 金鑰** | **密鑰** | 存取服務的通行證。**絕對不能放在網頁前端**，等於把鑰匙貼在門上。 |
| **localStorage** | **瀏覽器本機儲存** | 資料存在使用者自己的瀏覽器裡，換一台電腦就沒了。我們用它存作答草稿。 |
| **CDN** | **內容傳遞網路** | 從別人的伺服器載入共用資源（字型、圖示庫），加快速度。 |

## 四、品質與問題

| 你會看到的詞 | 中文說法 | 白話解釋 |
|---|---|---|
| **bug** | **程式錯誤** | 程式沒照預期運作。 |
| **debug** | **抓錯／除錯** | 找出並修掉錯誤。 |
| **regression** | **回歸（弄壞舊功能）** | 改 A 功能後不小心弄壞了 B 功能。「回歸測試」就是回頭確認沒弄壞。 |
| **dead code / 死碼** | **無用程式碼** | 留在檔案裡但永遠不會被執行的程式。佔空間、造成誤解、增加維護風險。 |
| **refactor** | **重構** | 整理程式碼的寫法，但**不改變功能**。 |
| **technical debt** | **技術債** | 為了求快而留下的爛攤子，之後要還。 |
| **P0 / P1 / P2 / P3** | **嚴重程度** | P0＝最嚴重、要馬上處理；P1＝重要；P2＝可以排隊；P3＝小問題。 |
| **XSS** | **惡意程式碼注入** | 有人在網址或輸入框裡塞一段程式，你的網站傻傻執行了。 |
| **權限提升** | （同左） | 本來沒權限的人，透過漏洞取得了不該有的權限。 |
| **console** | **瀏覽器主控台** | 開發者用來看錯誤訊息的地方（按 F12 可以叫出來）。 |

## 六、協作用語（Claude 與 Codex 之間）

| 你會看到的詞 | 白話解釋 |
|---|---|
| **工作包** | 一次任務的範圍定義：改哪些檔、不准改哪些、做完要達到什麼標準。 |
| **單一寫入者** | 同一個檔案同一時間只能有一個人改，避免互相覆蓋。 |
| **檔案占用表** | 記錄「現在誰在改哪個檔」的表，寫在 `PRD/原味藍圖_跨代理交接與檔案占用表.md`。 |
| **產品驗收 / 技術驗收** | 兩種不同的檢查。產品＝畫面對不對、流程順不順（Codex）；技術＝功能有沒有壞、資料安不安全（Claude）。兩邊都過才送你決定要不要上線。 |
| **保護清單** | 列出「碰了會壞掉」的程式碼，避免改外觀時弄壞評測功能。 |
| **已實作 ≠ 已驗證 ≠ 已發布** | 三個不同階段。「寫好了」不等於「測過了」，「測過了」不等於「上線了」。看到我們用這些詞時要分清楚。 |
