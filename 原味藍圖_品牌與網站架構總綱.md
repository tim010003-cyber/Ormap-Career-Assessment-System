## 1. 文件用途與使用方式

### 1.1 文件用途

本文件是「品牌決策」與「網站規劃／實作」之間的共同基準，目的如下：

1. 保存目前可由專案檔案直接確認的品牌文字、頁面、路由、工具狀態與使用者流程。
2. 防止後續任務把 mockup、推論、規劃中功能或舊名稱誤當成已定稿內容。
3. 讓內容、設計、前端、後端與驗收任務使用同一套狀態標籤和決策紀錄。
4. 當品牌定位或產品範圍仍不明確時，明確指出必須回到使用者討論的事項。

本文件不是一次性的分析報告，也不代表所有內容均已獲使用者核准。後續任務應先查閱「決策區」與「變更紀錄」，再決定可否直接實作。

### 1.2 任務權限邊界

網站任務可直接執行的範圍：

- 依「已定稿」內容修正明確的錯字、錯誤連結或狀態誤標。
- 接上已確認存在、且產品用途已確認的入口。
- 依現有設計系統補齊響應式、鍵盤操作、焦點樣式和減少動態支援。
- 在不改變品牌含義與產品範圍的前提下，修正跨頁顯示不一致。

必須回到使用者討論的範圍：

- 品牌定位、核心承諾、主要受眾或次要受眾。
- 六項工具的正式對外名稱、服務範圍和上線狀態。
- 首頁主要 CTA、登入門檻、頁面新增／刪除或公開範圍。
- 課程、顧問、諮詢、組織方案、定價、預約、金流或名單蒐集。
- 將 mockup 或 Demo 升格為正式產品。

### 1.3 狀態標籤

| 標籤 | 定義 |
|---|---|
| 已確認 | 可由使用者指示、正式檔案、路由或程式行為直接確認。 |
| 已定稿 | 檔案或使用者明確標示不可改寫；若不同來源互相衝突，仍須列入待決策。 |
| 暫定 | 可作為下一輪規劃基準，但尚未取得最終品牌／產品決策。 |
| 推論 | 根據已確認事實作出的合理解讀，不得冒充事實。 |
| 待使用者決策 | 會改變定位、範圍、主要 CTA 或公開承諾，必須由使用者決定。 |
| 尚未驗證 | 未透過瀏覽器、線上環境、實際使用者或外部服務確認。 |

### 1.4 更新規則

1. 每次變更品牌、工具名稱、頁面狀態或主要 CTA，必須同步更新本文件的決策區與變更紀錄。
2. 不得直接刪除舊決策；應保留「原內容、新決策、原因、影響頁面」。
3. mockup 的內容不得覆蓋正式頁現況，除非已有明確移植決策。
4. 規劃中或開發中功能不得因為存在文字、卡片或 Demo 就改標為已上線。
5. 未驗證的線上部署、外部連結、權限或商業流程必須維持「尚未驗證」。

## 2. 品牌核心

### 2.1 已確認

**品牌名稱**：原味藍圖 Ormap  
**目前正式首頁標題**：原味藍圖 Ormap｜生涯探索與人生設計  
**目前正式首頁一句話主張**：看懂自己的原味，建構想要的人生藍圖。

四個核心概念：

| 概念 | 已確認含義 |
|---|---|
| 原味 | 辨認反覆牽引自己的力量；理解特質、需求、價值、喜惡、天賦與既有優勢。不是固定本質。 |
| 清醒 | 閱讀所在的現實場域；研究機會、限制、資源、代價與仍待確認的資訊。 |
| 藍圖 | 形成暫時可行的設計；建立目標與策略，同時保留途徑、節奏與環境變化的調整空間。 |
| 勇敢 | 用承擔得起的行動取得真實回饋，再帶回下一輪理解與設計。 |

**品牌承諾（已確認自正式首頁）**：協助使用者整合特質、價值需求、能力與經驗，理解環境中的資源、限制、機會與風險，形成可執行、可驗證、可調整的人生方向。

**語氣原則（已確認自首頁與交接文件）**：

- 支持、清醒、具體。
- 不製造焦慮，不以恐懼推動轉換。
- 不替使用者做決定。
- 不把評測分數或類型當成固定人格結論。
- 不把「保留彈性」誤寫成沒有方向、反對計畫、純隨機或宿命論。
- 清楚說明工具協助覺察與選擇，不提供心理治療、醫療、法律或財務建議。

### 2.2 推論

- **推論：主要受眾**為正處於工作、生涯或角色轉折，想釐清需求、興趣、能力、生活配置與下一步的人。
- **推論：次要受眾**為主管、HR、企業主與生涯／職涯諮詢師。
- **推論：品牌差異**不在於提供單次測驗答案，而在於把自我理解、現實研究、策略設計與行動驗證連成循環。

### 2.3 尚待品牌決策

1. 「生涯探索與人生設計」是否為正式且長期的一句話定位。
2. 主要受眾應優先聚焦個人職涯轉折、整體生涯設計，或同時涵蓋組織端。
3. 「原味」應被定義為品牌隱喻、方法名稱，還是可被產品化的核心模型。
4. 首頁與會員系統是否全面改用「原味藍圖」，取代「Ormap SaaS／Ormap CRM」。
5. 組織端是同一品牌的次級服務線，還是需要獨立入口與子品牌。

## 3. 網站目標與角色

### 3.1 網站應協助訪客完成的任務

1. 理解原味藍圖的核心觀點與方法。
2. 辨認自己目前最需要處理的問題。
3. 選擇合適且確實可用的探索工具。
4. 完成評測、保存結果、回看報告與追蹤變化。
5. 把洞察轉成重測、行動、諮詢、課程或其他後續支持。
6. 讓主管、HR 與企業主找到清楚的組織端方案與聯絡方式。
7. 清楚理解登入、資料使用、授權、服務限制與隱私資訊。

### 3.2 各層角色

| 層級 | 角色 |
|---|---|
| 品牌網站 | 建立信任、解釋方法、呈現問題入口與工具／服務範圍。 |
| 探索工具 | 取得結構化自我資料，形成可閱讀的結果與反思。 |
| 個人中心 | 保存歷史、重測、比較變化、管理諮詢師授權。 |
| 報告與軌跡 | 把單次結果與長期變化轉成可理解的線索。 |
| 服務／課程／顧問 | 承接需要陪伴、應用或更深策略設計的使用者。現況尚未形成完整站內頁面。 |
| 組織端方案 | 將對人的理解轉成職務、人才標準與甄選流程。現況只有首頁介紹與未接線 Demo。 |

### 3.3 轉換目標

**暫定主要轉換**：選擇一項適合當下問題、且真實可用的探索工具。  
**暫定次要轉換**：Google 登入／回到個人中心。  
**推論之後續轉換**：查看報告、重測、諮詢、課程或組織方案。

> 待使用者決策：首頁主要 CTA 應先登入、先選問題、先選工具，或依使用者狀態動態切換。

## 5. 使用者旅程

### 5.1 新訪客理解品牌

```text
入口：網站根目錄
→ Hero
→ 理念
→ 建構方法
→ 工具區
→ CTA：選擇 TWA／HTI，或登入
```

結果：理解品牌並辨認可能的探索入口。  
斷點：Hero 三個不同措辭的 CTA 都只導向同一工具區；工具卡未說明登入門檻。

### 5.2 個人端選擇探索工具

目前可成功的正式流程：

```text
首頁「使用 Google 登入」
→ dashboard
→ 選擇 TWA 或 HTI
→ 工具介紹與準備
→ 作答
```

斷點：未登入者若直接點首頁工具卡，工具頁會把使用者送回首頁，沒有提示原因或接續登入。

### 5.3 完成工具、查看結果、採取下一步

```text
完成工具
→ 儲存 assessments
→ dashboard 歷史紀錄
├─ 查看報告
├─ 以此重測
├─ 授權／取消授權諮詢師
└─ TWA 兩筆以上 → 成長軌跡

報告
→ 閱讀洞察
→ CTA：與諮詢師深度對談
→ 外部 LINE
```

結果：得到單次報告、歷史紀錄與有限的下一步。  
斷點：站內沒有行動計畫、課程推薦、服務詳情、費用或預約流程。

### 5.4 尋找服務／課程／顧問

```text
首頁
→ 無服務／課程總入口
→ 完成 HTI 或打開歷史報告
→「與諮詢師深度對談」
→ 外部 LINE
```

結果：可到外部 LINE 聯繫。  
斷點：沒有諮詢師介紹、服務範圍、課程目錄、價格、預約或回站路徑。

### 5.5 組織端／主管／HR 尋找方案

```text
首頁工具區
→ 組織
→ 職務設計系統／課程／顧問配套
→ 無按鈕、無連結
```

→ 建立案例
→ Mock AI 整理
→ 人工確認

斷點：Demo 使用 localStorage、規則式 Mock AI 與假身分切換，不是真實產品權限或後端。

## 8. 內容與介面規則

### 8.1 品牌用語

1. 「原味」不可寫成固定本質、命定答案或不可改變的人格。
2. 「清醒」不可寫成放棄理想、壓抑需求或只接受現實。
3. 「藍圖」不可寫成一次決定終點或保證成功的計畫。
4. 「勇敢」不可寫成盲目豪賭、忽視風險或用焦慮推動行動。
5. 結果應描述線索、傾向、條件與待驗證假設，避免把人固定成標籤。

### 8.2 工具命名

- 每項工具應有一個正式對外名稱；專業縮寫可作副標，不應在不同頁面變成不同主名稱。
- 同一工具的首頁、dashboard、工具頁、報告和軌跡名稱應一致。
- 工具名稱變更屬品牌／產品決策，必須由使用者確認後才能全站替換。

### 8.3 狀態標籤

| 標籤 | 使用條件 |
|---|---|
| 已上線／可使用 | 有可到達路由、實際功能、必要權限和可完成結果。 |
| 免費／付費 | 只有在費用政策已確認時才可使用。 |
| 開發中 | 已有明確開發工作但尚不可對外完成。 |
| 規劃中 | 只有方向或需求，尚未進入可用開發狀態。 |
| Demo／概念稿 | 使用假資料、Mock AI、localStorage 或僅展示互動。 |
| 課程／顧問配套 | 必須另有可理解的服務說明或真實承接 CTA，否則可能造成誤導。 |

### 8.4 CTA 命名

- CTA 應直接描述下一步：例如「使用 Google 登入」「開始工作價值羅盤」「查看報告」「了解組織方案」。
- 同一區塊不應用三種文字導向同一目標，除非有明確層級理由。
- 需要登入時應在 CTA 前或按下後提供清楚說明，不應無提示送回首頁。
- 不得使用「發送報告」描述只有儲存、沒有寄送的行為。
- 規劃中功能不得出現仿正式操作的「開始」「立即使用」按鈕。

### 8.5 個人端與組織端分流

- 首頁第一段優先服務個人訪客；組織端置於個人探索之後是目前已確認的順序。
- 組織端應有清楚標題、對象、問題、交付內容與獨立 CTA。
- 內部 CRM／諮詢師後台不得作為公開組織服務頁使用。
- 若兩端共用品牌，導覽和頁尾仍需提供明確切換方式。

### 8.6 響應式與內容一致

- 手機與桌機應保留相同核心承諾、工具狀態、風險說明與主要 CTA。
- 可因版面縮短裝飾、次要說明、欄數或圖片比例，但不得在手機隱藏關鍵限制或登入門檻。
- 斷行可調整，文字不得因此被改寫成不同承諾。

### 8.6A 字體與閱讀節奏

- 對外頁面目前使用 Noto Sans TC 作為內文、Noto Serif TC 作為主要標題。
- 長段落應使用較寬鬆行距與適度字距，避免用縮小字級解決文字過多。
- 標題可以有輕微負字距，但不得讓中文字黏連或降低辨識度。
- 內文閱讀寬度、段落間距與卡片文字密度應在桌機與手機分別驗證。
- 現行逐區文字盤點與使用者校稿入口：`PRD/原味藍圖_公開網站文案盤點與調整表_v1_2026-07-21.md`。

### 8.7 可及性、導覽與信任

- 所有互動需可用鍵盤操作並有可見 focus。
- 圖片需有合適替代文字；純裝飾圖應正確標示為裝飾。
- 動畫需支援 `prefers-reduced-motion`。
- 每頁應有清楚的返回入口、目前位置或麵包屑。
- 登入、資料保存、諮詢師授權、外部連結和刪除操作需有清楚說明。
- 頁尾至少提供品牌、隱私、條款、回饋／聯絡和必要免責資訊。

## 10. 決策區

### 10.1 已定稿

1. `index.html` 檔案內明確標示的正式 Hero 文字，原文收錄於第 6.4 節；未有使用者新決策前不得擅自改寫。
2. 吸引子語意是「方向與目標可以清楚，抵達的路不只一條」，不得改成無終點、反計畫、純隨機或宿命論。
3. 品牌語氣應支持、清醒、具體，不製造焦慮，不替使用者決定。
4. 已上線工具必須比開發中／規劃中工具更醒目；未完成工具不得呈現為可操作假按鈕。
5. 組織端內容排列在個人探索之後，不與 Hero 爭奪第一層注意力。
6. Index 需要一個簡潔的「打開／展開」體驗，並以現有 Hero 吸引子圖片作為展開設計的一部分；效果應有記憶點但不複雜。

### 10.2 暫定

1. 公開首頁主要任務是協助訪客理解品牌並選擇探索入口。
2. 工作價值羅盤與何倫興趣與行動風格是目前兩項正式個人工具的對外名稱。
3. 個人端、整體生涯與組織端可置於同一品牌下，但需更清楚分流。

### 10.3 待使用者決策

1. 正式 Hero 與 `home-attractor.html` 哪一份是 canonical 文案。
2. 正式一句話定位、主要受眾、次要受眾與核心承諾。
3. 六項工具的正式名稱、費用、上線狀態與登入門檻。
4. 首頁主要 CTA 與登入前後策略。
5. 問題導向或分類導向的第一層資訊架構。
6. 個人諮詢、課程、顧問與組織方案的範圍、價格和承接方式。
7. 職務設計 Demo 的公開、隱藏或產品化方向。
8. 是否新增、刪除或合併頁面。
9. 是否將會員端與管理端全面改為「原味藍圖」新品牌。
