## 0. 任務總述

你要接手「原味藍圖 · 職務設計與甄選」這支 Web App，把它從一包**能跑但孤立的原型**，整合成**品牌一致、可以在課堂投影、可以放進網站動線**的成品。

工具本身的功能已經完成而且正確：從問題釐清一路到產出四份 Word 文件，三個示範案例都能完整跑完。**你的工作不是重做功能，是把它「收進來」並「做好看」。**

兩件事同時做：

- **整合**——目前它是三個各自為政的頁面，用自己的一套樣式，跟整個網站沒有任何連結。
- **視覺**——目前是工程稿的成熟度，還不能給客戶看。

## 1. 這支工具在做什麼（決定視覺該長什麼樣）

它不是「職務設計」工具，而是**問題解決導向**的思考工具。三段遞進：

1. **問題釐清**——你真正遇到的問題是什麼？（「缺人」只是假設，不是結論）
2. **職務需求界定**——過濾。只有真的該交給人才的才交給人才，其餘退回團隊或負責人。
3. **職務與人才規格**——這時才決定正職／接案／專案／承攬，以及什麼樣的夥伴適合。

**關鍵：它不是設計給一個人獨自填寫的。** 真實情境是兩個人、兩個角色：

- **回應者**（企業主／用人主管）看著主畫面思考與回答，他只該看到「現在這一題」；
- **引導提問者**（顧問／學員）看著右欄的提問腳本，一次問一題。

課堂上還會出現「兩個視窗」的用法：一個投影給大家看，一個是引導者自己的。所以主畫面要**在投影機上遠看讀得清楚**，而且**乾淨到沒有雜訊**——任何多餘的說明文字，對正在思考的人都是干擾。

設計目標一句話：**讓數位工具輔助大家在做思考時可以更輕鬆。**

## 4. 絕對不要動（動到會壞，且很難察覺）

- **任何 `.js` 檔案**與 `server/`。
- **HTML 裡的 `id`、`data-*` 屬性、表單 `name`、`<script type="module">` 區塊內的 JavaScript。** JS 大量依賴 `querySelector('#…')` 與 `dataset`，改名就靜默失效。
- **JS 會 `innerHTML` 寫入的容器**：`#moduleArea`、`#stageRail`、`#promptRail`、`#caseList`、`#caseFormFields`、`#statusCard`、`#formCard` 等。你可以改它們的 class 與外層包裝，**但不要改 id、也不要把它們搬到別的父層**。
- **`case.html` 的 `applyRailState()`**：它用 `style.setProperty(..., 'important')` 動態設定主畫面寬度，且**刻意沒有 CSS transition**——曾經加上 `transition: max-width` 導致寬度整個卡住。不要幫它加回過場動畫，也不要在 `#moduleArea` 上設 `margin: 0 auto`（grid item 用 auto margin 會吃掉剩餘空間，讓 `max-width` 失效）。
- **欄位文案、問題文字、階段名稱**：使用者定稿的產品內容，一律不得改寫。可以改排版，不能改字。
- **AI 提示詞與輸出契約**（`jd-ai.js` 的 `COMMON_RULES` 與信封形狀）。
- **產品名稱、定位、成熟度標示、價格**：這些是使用者的決定。`organizations.html` 現在明講「目前定位為課程／顧問配套，而不是已公開上線的自助工具」——**你新增的入口與 intro 頁必須維持這個口徑**，不可以寫成已上線的自助產品。

## 5. 要做的事

### 5.1 整合進網站

- 三頁套上 `brand.css` 與全站的 `site-header` / `site-footer` / 麵包屑（照 `tools.html`、`organizations.html` 的既有 markup 抄）。**注意**：工具內部大量使用 Tailwind utility class，`brand.css` 與 Tailwind 同時存在時要確認沒有互相覆蓋，必要時把工具內部的樣式收進 `jd-theme.css` 並限定作用範圍。
- 新增 `tools/job-design/jobdesign-intro.html`：說明這是什麼、適合誰、需要兩個人、大概要花多久、會產出什麼四份文件、目前的成熟度標示。比照 `tools/twa-intro.html` 的結構與語氣。
- `tools.html` 加入職務設計的入口卡，連到 intro 頁。
- `organizations.html` 的 `#job-design` 段落接上通往 intro 頁的連結。
- 把三份重複的 tailwind config 與 `<style>` 收斂成一份共用樣式。

### 5.2 補上說明與 onboarding

- 案例列表頁：第一次進來（沒有任何案例時）要看得懂這是什麼、該從哪裡開始。目前的空狀態只有一句「還沒有案例」。
- 工作區：讓引導提問者一眼知道右欄是給他看的、中欄是給回應者看的。
- 「載入示範案例」的三個選項要說明各自在示範什麼（尤其「權責矛盾」是刻意展示系統會擋下來，不是壞掉）。

### 5.3 視覺精修

品牌基調：**「先清醒，再勇敢」**。沉穩、支持、留白充足，不販賣焦慮、不用力推銷。沿用既有色票，**不要新增色系**：

| 用途 | 色票 | `brand.css` 變數 |
|---|---|---|
| 背景 | `#FBF8F2` | `--brand-paper` |
| 主文字／深底 | `#142033` | `--brand-ink` |
| 主色 | `#1E9188` | `--brand-teal` |
| 淺底 | `#DCECEF` | `--brand-mist` |
| 強調／警示 | `#E88A55` | `--brand-clay` |

字體：標題 `Noto Serif TC`，內文 `Noto Sans TC`。圖示：`lucide` 1.8.0。**不要引入任何新的 CSS 框架、字體、圖示庫或 npm 套件**——這個專案沒有 build step，全部靠 CDN 與靜態檔案。

優先順序：

1. **主畫面（中欄）的呼吸感**——一頁一題的節奏對了，但字級層次、行距、卡片留白仍偏工程感。讓「核心問題」真正成為視覺焦點，其餘退到背景。
2. **三欄的視覺分工**——左欄「階段」是導航、中欄「思考」是主體、右欄「提問」是給另一個人看的腳本。三者現在長得太像，應該一眼分得出主從。收合／展開的切換也要順。
3. **投影可讀性**——1280×720 投影、觀眾距離 3 公尺，主畫面的核心問題與選項要讀得到。必要時放大主畫面字級。
4. **狀態語彙一致化**——「待確認」「矛盾」「已完成」「AI 整理中」目前顏色與形狀不完全一致。統一成一套 badge／提示系統：矛盾用 clay、完成用 teal、中性用 ink 淡階。
5. **產出文件的預覽與下載區**——四份文件是使用者的成果，值得一個像樣的收尾畫面，目前偏樸素。
6. `index.html` 案例列表與 `settings.html` 設定頁比照同一套語彙收斂。

## 7. 給使用者的說明（不用貼給 Codex）

### 為什麼建議這樣切

你問的是「整包丟給 Codex，包含整合與視覺」。上面這份任務包涵蓋了**呈現層的整合**——品牌統一、網站動線、intro 頁、說明文案、視覺精修。這一整包 Codex 一次做完是合理的，因為它們互相牽動，拆開反而會來回打架。

但有一件事我**刻意留在這個任務包外面**：**帳號與雲端儲存**。

目前工具的資料存在瀏覽器的 localStorage，意思是換一台電腦、換一個瀏覽器，案例就不見了，也無法分享給別人。要解決這件事必須接上 Firebase Auth 與 Firestore，而那會動到：

- `firestore.rules`（存取控制）
- 使用者身分與授權語意
- 現有登入動線與 Dashboard

依照 `AGENTS.md` 與跨代理協作契約，Auth／Firestore／Rules 屬於我（Claude）的職責範圍，而且缺陷登記簿裡目前還有**未解決的 P1 存取控制風險**。把它併進一個以視覺為主的任務包，風險不成比例——視覺改壞了看得出來，權限改壞了看不出來。

好消息是 `jd-store.js` 當初就是為了這件事抽象出來的，介面（`listCases` / `getCase` / `saveField` / …）不用改，之後換掉實作即可。所以**先做呈現層整合完全不會擋到後面接後端**。

### 建議順序

1. **現在**：Codex 執行上面這包（整合＋視覺）。
2. **之後**：你實際投影試用、確認可以上課，產品驗收。
3. **再之後**：如果確定要讓客戶自己在線上使用，開一個獨立的後端工作包（Auth + Firestore + Rules），由我來做，並補上必要的安全性回歸測試。

### 一個要留意的地方

`.github/workflows/firebase-hosting-pull-request.yml` 的註解說 `tools/job-design/**` 被忽略，但實際的 `firebase.json` 忽略清單裡**沒有**這一條——實際設定為準。也就是說**這些檔案一旦提交，就會出現在 PR 預覽，合併後也會出現在正式站上**。所以 §4 才特別要求維持 `organizations.html` 現有的「課程／顧問配套，非公開自助工具」口徑，避免還沒定案就被當成已上線產品。
