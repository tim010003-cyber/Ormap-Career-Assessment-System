## 1. 開始工作前必讀

依序確認以下文件；若任務只涉及單一模組，仍不得略過共同治理、缺陷與占用狀態：

1. 本文件 `AGENTS.md`。
2. `PRD/README.md`，先依任務選擇需要的產品、文案、工程或歷史文件。
3. `PRD/原味藍圖_跨代理協作契約_v1_2026-07-20.md`。
4. `PRD/原味藍圖_已知缺陷與風險登記簿.md`。
5. `PRD/原味藍圖_跨代理交接與檔案占用表.md`。
6. 若涉及評測、報告、時間線或白板：`PRD/原味藍圖_評測與白板介面保護清單_v1_2026-07-20.md`。
7. `原味藍圖_品牌與網站架構總綱.md`。
8. `PRD/原味藍圖_評測優先平台轉型與分階段開發藍圖_v1_2026-07-20.md`。
9. 若涉及 TWA／HTI 公開內容或內外頁拆分：`PRD/原味藍圖_TWA與HTI架構與文案盤點_v1_2026-07-21.md`。
10. 與當次模組直接相關的 PRD、規格與交接紀錄。
11. `CLAUDE.md` 可作為既有技術架構與 Claude 工作提示，但不得凌駕使用者最新指示、專案共同文件或當次工作契約。

若文件互相衝突，先停止衝突範圍的寫入，列出「原文、來源、影響與需要誰決定」，交由使用者裁決；不得自行選一份當成定稿。

## 2. 決策優先序

1. 使用者在當次任務中的最新明確指示。
2. 使用者已確認的品牌、產品、上線與資料決策。
3. 品牌與網站架構總綱、已確認 PRD 及決策紀錄。
4. 跨代理協作契約與當次工作包。
5. 代理專用說明、程式註解與歷史文件。
6. 代理的推論或偏好。

較新的文件不會自動推翻較舊但已定稿的決策；變更必須記錄原內容、新決策、日期、原因與影響範圍。

## 3. 中立角色模型

每個工作包都要明確指定角色，不以代理名稱推定最終權力：

- **使用者／產品決策者**：決定品牌定位、受眾、產品範圍、工具名稱、成熟度、價格、主要 CTA、頁面增刪與正式發布。
- **產品與體驗負責者**：負責資訊架構、使用者流程、內容模型、UI／UX、設計系統及產品驗收。
- **工程與營運負責者**：負責資料、Auth、Firestore、權限、安全、CI／CD、部署與技術驗收。
- **模組負責者**：負責被指派模組的實作、單元驗證與交接。
- **發布執行者**：只在必要驗收與發布核准完成後執行部署；執行部署不等於擁有產品決策權。

目前的能力導向預設分工記載於協作契約；它是預設安排，不是任何代理對全專案的永久所有權。當次任務可由使用者另行指定。

## 4. 工作開始閘門

任何寫入前，至少確認並留下：

- 任務目標與本輪不處理事項。
- 可修改、可新建及禁止修改的完整路徑。
- 已讀缺陷登記簿，並確認本次是否接觸未修復項目。
- 已在檔案占用表確認沒有衝突，並登記唯一寫入者與工作包。
- 工作包負責者、必要審查者與發布執行者。
- 是否允許覆寫、移動、重新命名、刪除、安裝、外部呼叫或部署。
- 資料、登入、權限、金流、通知與外部服務是否受影響。
- 若要求登入後驗證：指定預覽網址、測試身分取得方式、允許的瀏覽器／登入操作、可否建立／修改／刪除測試資料，以及未能執行時由誰補測。使用者已同意代理可在指定預覽環境做登入後前臺測試，但工作包仍須提供確切目標與資料操作邊界；不得讀取、傳遞或記錄密碼、token、cookie 或其他憑證。
- 驗收標準、回歸範圍與尚未驗證項目。

資訊不足時可以繼續唯讀盤點，但不得猜測後寫入。

## 5. 單一寫入者與檔案邊界

- 同一時間，同一檔案只能有一個寫入者。
- 寫入前必須在 `PRD/原味藍圖_跨代理交接與檔案占用表.md` claim 具體路徑；未登記或無法確認即時狀態時，只可唯讀。
- 工作開始前先查看工作區狀態；不覆蓋、不還原、不整理其他代理或使用者的變更。
- 跨模組修改要先列出介面、輸入、輸出、資料路徑、權限、穩定 ID／DOM hook 與驗收方式。
- 若必要變更會碰到別人正在修改的檔案，先停止該檔案，改以交接或排程方式處理。
- 未經明確授權，不得批次重構、全站改名、搬移檔案、改變資料格式或擴大部署範圍。

## 6. 既有功能保護

以下屬高回歸風險區，任何外觀、結構或共用樣式變更都必須保護其行為並完成相應驗證：

- TWA／HTI 的題目、計分、結果映射與 JSON 資料來源。
- 草稿、自動儲存、恢復、完成與錯誤狀態。
- Firebase Auth、Firestore 讀寫、資料隔離與權限規則。
- `report.html`、`timeline.html`、`whiteboard.html` 的資料與跨頁流程。
- whiteboard mirror mode 及其相依的 class、狀態旗標與 DOM hook。

評測、報告、時間線與白板的首批穩定掛點記錄於 `PRD/原味藍圖_評測與白板介面保護清單_v1_2026-07-20.md`。該文件目前仍待 Claude 技術確認；確認前不得進行涵蓋這些掛點的跨結構重寫或全域 CSS 重構。

這些檔案不是任何代理的永久禁區；能否修改取決於當次工作契約、檔案所有權與回歸證據。只改「樣式」也不能假設不影響行為。

## 7. 品牌、內容與狀態規則

- 使用繁體中文維護對外內容與產品文件。
- 清楚區分：`已確認`、`推論`、`假設`、`提案`、`待使用者決策`、`已實作`、`已驗證`、`尚未驗證`。
- 不以單獨的「已完成」概括交付。至少分開回報：`寫入權`、`實作`、`Git`、`驗證`、`可見環境／發布`；若沿用「完成」一詞，必須附完整限定，例如「實作完成／未提交／部分驗證／僅工作區可見」。工作包的最終交付完成由使用者確認。
- Demo、mockup、存在路由或註解中的規劃，不等於已上線。
- 開發中、規劃中及配套服務不得呈現為可立即操作。
- 未經使用者確認，不得改寫品牌定位、Hero 定稿文案、工具名稱、價格、服務範圍或主要 CTA。
- 正式 `index.html` 與 `mockups/home-attractor.html` 的 Hero 來源仍有衝突；在使用者裁決 canonical 版本前，不得自行合併或宣告其中一份取代另一份。

## 8. 安全與敏感資訊

- 不讀取、輸出或提交 `.env`、密鑰、token、cookie、憑證、私鑰或個資。
- 已知安全與營運問題統一記錄於 `PRD/原味藍圖_已知缺陷與風險登記簿.md`；缺陷登記不等於授權修復或部署。
- 不把授權判斷、秘密值、付費 API 呼叫或管理權限放入前端。
- 涉及 Auth、Firestore Rules、個資、金流、EDM、Google 整合或外部表單時，先定義資料目的、最小欄位、同意、權限、錯誤處理與部署方式。
- 未經明確授權，不呼叫付費服務、不發訊息、不送出表單、不部署正式環境。

## 9. Git、預覽與發布

- 不直接以未審查變更更新正式環境。
- 原則上使用獨立分支、可閱讀的差異、PR 與預覽環境；若現況做不到，將其列為發布阻擋或由使用者明確接受風險。
- 實作者可把範圍內成果提交到任務專用的非 `main` 分支，以建立可回復、可審查的保存點；不得 blanket-stage 或混入其他工作包。推送、開 PR 與建立預覽須符合工作包的發布方式，並指定預覽執行者；預覽執行者可由實作者、工程與營運負責者或其他指定代理擔任。
- 合併前至少完成與變更範圍相稱的產品驗收及技術驗收。
- 現行 `.github/workflows/firebase-hosting-merge.yml` 在 `main` 收到新提交時部署 live Hosting。因此，**合併到 `main` 就是正式發布動作**；使用者核准必須發生在合併前，不存在合併後、上線前的第二道確認關卡。
- 影響 Firestore 資料或權限時，程式碼、Rules、索引及實際部署狀態必須一併核對。
- `mockups/`、Demo、內部頁與測試資料不得因 Hosting 設定而被誤認為正式產品。
- 是否發布由使用者或其明確授權者決定；發布執行者只負責依核准內容操作與回報。
- 已確認且仍可達的 P0 預設阻擋受影響的預覽與正式發布；若預覽共用正式 Firebase、Auth 或資料，也不得自動視為隔離環境。P1 不自動阻擋所有發布，但每次發布前須逐項記錄是否受影響、是否擴大暴露、緩解措施、負責者、期限與使用者風險接受。缺陷狀態互相矛盾或證據不足時，受影響範圍不得發布，先完成狀態釐清。

## 10. 完成交接

每次交付至少回報：

1. 本輪完成內容。
2. 修改／新建檔案清單。
3. 明確未修改的重要區域。
4. 驗證方法與結果。
5. 尚未驗證項目。
6. 已知風險、回復方式與建議下一步。
7. 若需另一代理接手：工作包、檔案所有權、介面契約、驗收結果及剩餘阻擋。
8. 更新缺陷登記簿與檔案占用表：新增／變更狀態、實際交還與下一位可開始的條件。

不得以「已完成」代替證據；也不得因測試通過就自行宣稱品牌、產品或上線決策已定稿。
